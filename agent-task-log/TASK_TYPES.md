# Task Type 枚举说明

## 1. 概述
统一的 Task Type 标签用于标记每份 Agent 任务日志的性质，确保文件命名、标题与索引保持一致。本文件列出允许使用的取值，并说明适用场景与排他规则。

## 2. 枚举列表
### 2.1 plan
- 适用于方案设计、技术调研、可行性分析、架构评估等“规划类”任务。

### 2.2 implementation
- 用于功能开发、落地实施或大范围的代码改动，覆盖新增特性、模块化重构落地等。

### 2.3 bugfix
- 针对缺陷定位、修复与回归验证的任务。通常伴随问题复现、修复方案与验证结果。

### 2.4 refactor
- 表示结构性调整或重构任务，强调改善代码结构、模块职责或可维护性，而非直接交付新功能。

### 2.5 optimization
- 用于性能、体验或资源使用优化，包括基准测试、瓶颈排查与优化措施落地。

### 2.6 analysis
- 面向复盘、评估、度量或状态报告等分析类任务，输出通常为评估结果与建议。

### 2.7 testing
- 覆盖测试计划、测试执行、测试工具改造与测试体系建设等工作。

### 2.8 documentation
- 专注于文档整理、规范梳理、知识库维护等，产出主要以文档为核心。

### 2.9 maintenance
- 面向运维、依赖升级、环境配置、CI/CD 调整等日常维护行动。

## 3. 使用规则
- 每份日志必须选择上述枚举中的**单一**类型，应用在文件名、标题与内容元信息中。
- 枚举取值需要保持英文小写，便于检索与自动化脚本处理。
- 如遇无法归类的新场景，先按最接近的类型处理，并在本文件记录补充计划。

## 4. 更新流程
1. 确认新增类型的必要性及对现有规范的影响。
2. 征得项目维护人或协作伙伴认可后更新本文件，并同步调整模板、README 等规范文档。
3. 批量更新历史日志，确保旧文件与新类型保持一致。
