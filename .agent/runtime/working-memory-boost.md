# Working Memory Boost

> 工作记忆持久化：跨 session 不丢失上下文

---

## 何时触发

**立即做**：用户说"存档"、"完成"、"好"

**自动做**：
- 改了 >3 个文件，或写了 >1000 tokens
- 出现明确决策（"选X"、"定了"）
- 解决阻塞

**不做**：
- 纯探索，无结论
- 仅改 typo/格式
- 测试失败
- 用户说"不用存档"

## 节点标记（轻量级）

> 比 CP 更细的粒度，记录关键思考节点，后期再降噪

### AI 自动标记（语义判断）

遇到以下场景，立即在 ongoing/*.md 追加「节点记录」：

| 信号类型 | 示例 | 记录内容 |
|----------|------|----------|
| **新发现/洞察** | "我突然想到..."、"关键洞察是..." | 时间 + 洞察摘要 + 来源位置 |
| **思路转折** | "不对，应该换个方向..."、"重新思考..." | 时间 + 转折原因 + 新思路 |
| **关键图/代码** | ASCII 图、Mermaid、代码块 >20 行 | 时间 + 图/代码摘要 + 对话位置 |
| **决策形成** | "那就定这个方案"、"优先级确定了" | 时间 + 决策内容 + 备选方案 |
| **待确认/回文** | "这个后面再验证"、"Phase 2 再优化" | 时间 + 待办项 + 预计阶段 |

### 用户显式标记

用户说以下关键词，立即记录前后上下文：
- "记下来"、"这是重点"、"mark一下"
- "看图"、"看这段代码"
- "这是关键转折点"

### 节点记录格式

在 ongoing/*.md 的当前 Phase 下追加：

```markdown
### 节点记录（2026-04-13）

| 时间 | 类型 | 内容 | 位置 |
|------|------|------|------|
| 04:30 | 图 | 五层运行架构草图（CLD→FCM→D2D） | [对话](link) |
| 04:35 | 决策 | 四图优先级待确认（业务/工程/流程/数据） | - |
| 04:40 | 洞察 | 上下文容易丢失，需加强记录粒度 | [对话](link) |
```

### 降噪原则

- **先记录，后整理**：宁可多记，不要漏记
- **每周回顾**：合并冗余节点，提炼为正式决策/调研
- **阶段归档**：Phase 结束时，将节点历史压缩为 Checkpoint

## 操作流程

### Step 0：读取看板

所有操作必须先读取 `.working-memory/board.md`，向用户展示：
- 进行中任务列表
- 最近归档（前 3 个）
- 最新数据分析和 aha-moment 链接

### Step 1：确认动作

| 触发词 | 动作 |
|--------|------|
| 存档、记录、同步 | `checkpoint` |
| 读档、恢复、继续 | `resume` |

默认 `checkpoint`。

### Step 2：完整性检查

**L1**：board 列的任务 ↔ ongoing/ 文件同步，缺失的尝试从 Issue 恢复
**L2**：本地文件 vs `.checkpoint` 对比 mtime，冲突时问用户
**L3**：对比 `HEAD` 与 `checkpoint.git_ref`，偏离时警告

### Step 3：执行动作

#### 存档（checkpoint）

流程：
1. 读取本地 `ongoing/任务名.md`
2. 更新 Checkpoint 状态表（用户或 AI 已更新）
3. 更新 `.checkpoint`：
   ```json
   {
     "issue": 15,
     "phase": "research",
     "completed": ["CP-18"],
     "next": "CP-19: 工程架构设计",
     "git_ref": "a1b2c3d",
     "timestamp": "2026-04-12T18:00:00Z"
   }
   ```
4. 同步 `board.md`：更新该任务的「状态」「下一动作」「更新」列
5. 写入远程 Issue（正文 + 评论）

远程层极简：
- **Issue 正文**（5 行内）：
  ```
  综合集成法架构洞察 - CP-18 完成
  Status: 🟡 research
  Next: 工程架构设计
  Local: .working-memory/ongoing/2026-04-10-02_...
  Git: a1b2c3d
  ```
- **评论**：仅记录关键节点（CP 完成、阻塞、决策）

#### 读档（resume）

**判断模式：**

| 条件 | 模式 | 操作 |
|------|------|------|
| 本地文件存在且 mtime > 远程 | 热恢复 | 读取本地，验证 L3 |
| 本地文件缺失或 mtime < 远程 | 温恢复 | 从 Issue 重建本地 |
| 用户显式指定 `#issue` | 强制重建 | 从指定 Issue 重建 |

**温恢复流程：**
1. `gh issue view <id> --comments`
2. 解析 Issue 正文 + 最新评论
3. 重建 `ongoing/任务名.md`（精简三区块格式）
4. 重建 `.checkpoint`
5. 更新 `board.md`（如该任务未列出则添加）
6. 输出恢复摘要

### Step 4：后置完整性检查（L3）

- 再次验证 Git 状态
- 对比本次操作前后的 `.checkpoint` 变化
- 确认远程同步成功（可选：抽查 Issue 正文）

## 文档格式规范

### board.md（全局看板）

```markdown
# 工作记忆看板

## 进行中

| 任务 | 阶段 | 状态 | 下一动作 | 更新 |
|------|------|------|----------|------|
| [#15] 综合集成法架构 | research | 🟡 | 工程架构设计 | 04-12 |

## 最近归档

| 任务 | 结果 | 日期 |
|------|------|------|
| [#14] 项目方向对齐 | ⬜ | - |

## 数据分析

- [Bug 健康度](./analysis/README.md)

## Aha Moments

- [模型选择](../aha-moments/2026-...md)
```

### ongoing/*.md（单个任务）

**最佳实践（阶段驱动结构）：**

```markdown
# Issue #{n}: {任务标题}

> 一句话目标：{简洁描述}

---

## Phase {m}: {阶段名称}（{时间范围}）{状态图标}

**阶段目标**：{本阶段要达成什么}

### Checkpoint

| CP | 内容 | 状态 |
|----|------|------|
| {n} | {具体内容} | ✅ |

### 关键上下文

- **决策**：{本阶段核心决策}
- **关键洞察**：{重要发现/认知}
- **技术选型**：{锁定/待定的技术栈}
- **风险备忘**：{已知风险}

### 新增决策

| 日期 | 决策 | 来源 |
|------|------|------|
| {YYYY-MM-DD} | {具体决策} | {来源} |

### 调研产出

| 主题 | 文档 |
|------|------|
| {主题} | [{文件名}](issue-{n}-sub_docs/{file}.md) |

### 实施产出

- [x] {已完成项}
- [ ] {待办项}

### 阶段关联文档

- `{path}`（说明）

### 回文/Pending

| 项目 | 说明 | 预计处理 |
|------|------|----------|
| 🔴 {阻塞项} | {说明} | {阶段} |
| 📝 {优化点} | {说明} | {时机} |

---

## Phase {m+1}: {下一阶段}（{时间范围}）{状态图标}

...

---

## 项目锁定

| 项 | 内容 |
|----|------|
| **项目名称** | {名称} — {一句话描述} |
| **EN Description** | {英文描述} |
| **CN Description** | {中文描述} |
```

**结构原则：**
1. **阶段为主线**：Phase 1 → Phase 2 → Phase 3，时间线清晰
2. **每个阶段内包含**：Checkpoint、决策、调研、实施、关联文档、回文
3. **内容完整保留**：所有洞察、备忘、决策原样记录，仅改变组织方式
4. **图标系统**：
   - ✅ 已完成 | ⏳ 进行中 | ⬜ 待启动
   - 📋 待办 | 🔴 阻塞 | 📝 回文/优化

## 一行回执

**存档：**
```
已存档 #15 | research 🟡 | Next: 工程架构设计
本地: .working-memory/ongoing/...md ✅ | Git: a1b2c3d ✅
```

**读档（热）：**
```
已恢复 #15 | 来源:本地 | research 🟡 | Next: 工程架构设计
```

**读档（温）：**
```
已恢复 #15 | 来源:远程 | research 🟡 | Next: 工程架构设计
重建: .working-memory/ongoing/...md | Git: 一致
```

## Edge Cases

- `gh` 未登录 → 提示登录后停止
- board/文件不一致 → 自动同步
- 时间戳冲突 → 问用户选哪个
- Git 偏离 → 警告并显示差异
- Issue 正文过长 → 压缩为 5 行
