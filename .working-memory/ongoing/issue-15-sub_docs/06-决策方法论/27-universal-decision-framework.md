# 通用决策框架（抽象层）

## 设计原则

**框架先于应用，抽象先于具体。本层定义通用决策规则，与任何具体项目无关。**

---

## 核心模型

### 域分类模型（Cynefin-Based）

所有决策情境分为四域，判定基于**因果关系的可知性**。

| 域 | 因果可知性 | 决策特征 | 时间特征 |
|----|-----------|----------|----------|
| **Clear** | 事前已知 | 标准化执行 | 即时 |
| **Complicated** | 分析后可知 | 专家判断 | 短期（天级） |
| **Complex** | 事后才知 | 涌现适应 | 中长期（周/月级） |
| **Chaotic** | 不可知/崩溃 | 紧急响应 | 立即（分钟级） |

### 判定流程

```
情境输入
   │
   ├─ 因果已知？→ Clear
   │
   ├─ 分析可知？→ Complicated
   │
   ├─ 事后才知？→ Complex
   │
   ├─ 需立即行动防崩？→ Chaotic
   │
   └─ 无法判定？→ 默认Complex（保守原则）
```

## 通用规则集

### 规则A：域适配执行（Domain-Adaptive Execution）

**A.1 Clear域规则**

```
原则：零决策
定义：执行者不做出任何选择，全部按预设执行。

要求：
1. 决策点前移至设计期
2. 执行期只允许多选一（Single Path）
3. 变体输入被拒绝或标准化转换
4. 输出格式严格固定

检查点：
□ 输入是否匹配预设格式
□ 流程是否完全线性（无分支）
□ 输出是否通过Schema验证
```

**A.2 Complicated域规则**

```
原则：多方案+延迟承诺
定义：必须对比多方案，决策前有强制等待期。

要求：
1. 至少3个方案并行评估
2. 决策动议后强制冷却期（建议72小时）
3. 必须收集反对意见
4. 关键决策需外部验证

检查点：
□ 是否有3个方案对比文档
□ 是否已过冷却期
□ 是否有反对意见记录
□ 是否有外部验证记录
```

**A.3 Complex域规则**

```
原则：探测优先+放弃预测
定义：通过小规模实验验证假设，禁止长期详细计划。

要求：
1. 任何假设需实验验证后才能扩大投入
2. 计划范围限制（建议6周内）
3. 计划粒度限制（建议粗于周）
4. 建立早期信号识别机制

检查点：
□ 是否完成探测实验
□ 计划是否在时限内
□ 是否定义早期信号指标
□ 是否有快速响应预案
```

**A.4 Chaotic域规则**

```
原则：先行动+分层响应
定义：不分析先行动，阻止恶化优先于解决问题。

要求：
1. 识别后强制时限内行动（建议15分钟）
2. 分层响应机制
   - L1：自动止损（熔断/降级/关闭）
   - L2：人工稳定核心功能
   - L3：根因分析与修复
3. 事后强制复盘

检查点：
□ 是否在时限内启动行动
□ L1是否自动触发
□ L2是否有人介入
□ 复盘是否在24小时内完成
```

### 规则B：分层自治（Layered Autonomy）

**分层结构**

```
战略层（Strategy Layer）
├── 时间尺度：月级
├── 决策内容：方向、边界、里程碑
└── 输出形式：目标+约束条件

战术层（Tactical Layer）
├── 时间尺度：周级
├── 决策内容：模块设计、资源分配、依赖协调
└── 输出形式：方案+接口契约

反应层（Reactive Layer）
├── 时间尺度：秒/分级
├── 决策内容：执行、数据处理、异常响应
└── 输出形式：结果+状态
```

**层间交互原则**

```
1. 层内自治：各层独立决策，不跨层实时协调
2. 契约优先：层间通过明确契约交互
3. 变更隔离：下层内部变更不得破坏上层契约
4. 信息向上：下层状态汇总向上，不细节汇报
```

### 规则C：重心分配（Focus Allocation）

**分类原则**

| 类型 | 定义 | 特征 | 资源分配 |
|------|------|------|----------|
| **Type 1** | 计算型 | 有明确最优解，可自动化 | 默认流程，快速通过 |
| **Type 2** | 分析型 | 多方案权衡，需专家判断 | 分析时间，对比评估 |
| **Type 3** | 战略型 | 定义问题本身，框架选择 | 80%资源，深度研讨 |

**三问筛选法（判定Type 3）**

```
1. 此决策现在不定，系统会立即停滞？
2. 此决策错误，6月后发现，代价不可承受？
3. 此决策正确，能带来数量级（10x）收益？

任一"是" → Type 3
全部"否" → Type 1或2
```

---

### 规则D：人机分工（Human-AI Division）

**分工矩阵**

| 域 | Type | 人类角色 | AI角色 | 协作模式 |
|----|------|----------|--------|----------|
| Clear | 1 | 设计契约 | 执行契约 | 人验收 |
| Complicated | 2 | 选择框架 | 填充选项 | 人决策 |
| Complex | 3 | 定义问题 | 挑战假设 | 共演化 |
| Chaotic | 任意 | 接管L2-L3 | 执行L1 | 紧急切换 |

**禁止行为**

```
□ Type 3决策禁止降级为Type 1/2处理
□ Type 1决策禁止占用Type 3资源
□ 人类禁止在Clear域干预执行细节
□ AI禁止在Complicated/Complex域做最终决策
```

---

### 规则E：冲突解决（Conflict Resolution）

**优先级（降序）**

```
1. 域判定规则（E > D > C > B > A）
2. Chaotic域覆盖（危机时优先）
3. 分层原则（层间冲突时）
4. 重心分配（资源冲突时）
5. 人机分工（责任不清时）
6. 域内规则（同域冲突时）
```

**默认策略**

```
域不明确 → 默认Complex域
类型不明确 → 强制Type 3处理
规则不明确 → 暂停决策，升级处理
```

## 应用适配指南

### 适配步骤

```
步骤1：情境映射
├── 将具体情境映射到四域
├── 识别时间尺度
└── 确定决策类型（1/2/3）

步骤2：规则选择
├── 根据域选择A.1-A.4
├── 根据层级应用规则B
├── 根据类型应用规则C
└── 确定人机分工（规则D）

步骤3：定制检查点
├── 保留通用检查点
├── 添加领域特定检查点
└── 定义量化阈值

步骤4：建立反馈
├── 定义域间转换信号
├── 建立规则更新机制
└── 设置复盘周期
```

### 定制示例（非绑定）

**示例：软件系统开发**

```
Clear域定制：
├── 输入：代码提交触发CI/CD
├── 流程：固定的构建-测试-部署流水线
├── 输出：通过/失败的状态报告
└── 检查点：单元测试100%通过？构建无错误？

Complicated域定制：
├── 方案：架构A/B/C对比
├── 冷却期：72小时
├── 外部验证：架构评审委员会
└── 检查点：是否有负载测试对比？是否有安全评审？

Complex域定制：
├── 探测：2周MVP实验
├── 计划：6周滚动规划
├── 信号：用户留存率、NPS、性能指标
└── 检查点：实验假设验证？信号是否突破阈值？
```

## 元规则（关于规则本身）

### 规则更新

```
触发条件：
1. 每6周定期评审
2. 任何Chaotic事件后24小时内
3. 连续3次规则冲突无法解决时

更新流程：
1. 识别失效场景
2. 判断是否新增域类型
3. 修订或新增规则
4. 测试新规则在边界案例的有效性
5. 更新检查清单
6. 通知所有决策者
```

### 框架局限

```
本框架假设：
1. 决策者能相对准确判定域类型
2. 系统有基本的分层结构
3. 人机能力边界相对清晰

本框架不适用：
1. 完全无结构的混乱系统
2. 决策者有严重认知偏差无法自察
3. 资源约束极端到无法执行检查点
```
