# 决策策略层

## 定位

**策略层是将哲学转化为可执行规则的中间层。**

- 不解释"为什么"
- 只规定"怎么做"
- 提供检查清单和量化标准

---

## 策略1：域判定（必做第一步）

### 判定流程（30秒完成）

```
情境输入
   │
   ▼
问题1：因果关系已知？
├── 是 → Clear域 → 策略2
└── 否 → 继续

问题2：分析后可知？（专家能解）
├── 是 → Complicated域 → 策略3
└── 否 → 继续

问题3：事后才知？（需实验验证）
├── 是 → Complex域 → 策略4
└── 否 → 继续

问题4：需立即行动防恶化？
├── 是 → Chaotic域 → 策略5
└── 否 → 默认Complex域（保守）
```

### 判定示例

| 情境 | 判定 | 依据 |
|------|------|------|
| 数据格式转换 | Clear | 输入X→输出Y确定 |
| 架构选型 | Complicated | 需分析对比多方案 |
| 新市场进入 | Complex | 事后才知是否成功 |
| 服务器宕机 | Chaotic | 需立即止损 |

## 策略2：Clear域执行

### 核心策略：零决策

**定义**：执行者不做出任何选择，全部按预设执行。

### 执行标准

| 维度 | 要求 | 检查点 |
|------|------|--------|
| 输入 | 严格格式，拒绝变体 | □ 输入通过Schema验证 |
| 流程 | 单一路径，无分支 | □ 流程图无决策菱形 |
| 输出 | 固定格式，不变形 | □ 输出通过Schema验证 |
| 错误 | 显式报错，不猜测 | □ 错误信息可定位 |

### 禁止行为

```
□ 执行期动态调整参数
□ 接受非标准格式输入
□ 输出格式灵活适配
□ 隐藏错误或自动修复
```

### 度量指标

```
零决策率 = Clear域任务中无执行期决策的比例
目标：>99%

变异处理率 = Clear域任务中需人工处理变异的比例
目标：<1%
```

## 策略3：Complicated域执行

### 核心策略：多方案+延迟承诺

**定义**：必须对比多方案，决策前有强制等待期。

### 执行标准

| 阶段 | 要求 | 检查点 |
|------|------|--------|
| 方案生成 | ≥3个可行方案 | □ 方案文档完整 |
| 分析评估 | 各方案优缺点量化 | □ 评估矩阵完成 |
| 冷却期 | ≥72小时 | □ 时间戳验证 |
| 反对收集 | ≥1个有效反对意见 | □ 反对意见记录 |
| 外部验证 | ≥1位专家确认 | □ 专家反馈记录 |
| 决策记录 | 决策+理由+反对回应 | □ 文档归档 |

### 冷却期日历

```
Day 0：问题识别，动议发起
Day 1-2：AI/团队准备材料
Day 3-4：审阅+专家咨询
Day 5：最终决策
Day 6+：更新文档，冻结配置
```

### 禁止行为

```
□ 单一方案直接决策
□ 冷却期内强行通过
□ 无反对意见记录
□ 无专家外部验证
```

### 度量指标

```
多方案率 = Complicated决策中≥3方案的比例
目标：100%

冷却遵守率 = 实际等待/规定等待的比例
目标：≥100%

专家验证率 = 有外部专家确认的决策比例
目标：100%
```

## 策略4：Complex域执行

### 核心策略：探测优先+放弃预测

**定义**：通过小规模实验验证假设，禁止长期详细计划。

### 执行标准

| 维度 | 要求 | 检查点 |
|------|------|--------|
| 假设验证 | 实验后才扩大投入 | □ 实验报告完整 |
| 计划范围 | ≤6周 | □ 计划时间戳验证 |
| 计划粒度 | ≥周级 | □ 无日级详细任务 |
| 信号识别 | 定义3个早期指标 | □ 指标可观测 |
| 快速响应 | 信号出现后48小时内行动 | □ 响应记录完整 |

### 探测实验模板

```
实验名称：
假设：
实验规模：≤总预算10%
实验周期：≤2周
通过标准：
├── 定量指标：
└── 定性判断：

实验结果：
├── 假设验证？
├── 意外发现？
└── 下一步：
    ├── 假设成立 → 扩大投入
    ├── 假设部分成立 → 调整再实验
    └── 假设不成立 → 放弃或转向
```

### 滚动规划格式

```
当前周期：[日期] 至 [日期+6周]

Week 1-2：探测
├── 假设：
├── 实验：
└── 产出：

Week 3-4：响应
├── 信号观察：
├── 调整措施：
└── 产出：

Week 5-6：固化/再探测
├── 假设验证结果：
├── 是否固化到Clear域：
└── 下一周期假设：
```

### 禁止行为

```
□ 未经实验直接大规模投入
□ 制定>6周详细计划
□ 计划粒度细于周
□ 信号出现后无响应
```

### 度量指标

```
探测覆盖率 = Complex决策前有实验的比例
目标：100%

计划合规率 = 计划≤6周且粒度≥周的比例
目标：100%

响应及时率 = 信号后48小时内响应的比例
目标：>90%
```

## 策略5：Chaotic域执行

### 核心策略：先行动+分层响应

**定义**：不分析先行动，阻止恶化优先于解决问题。

### 执行标准

| 层级 | 时限 | 行动 | 责任人 |
|------|------|------|--------|
| L1 自动 | 0-15分钟 | 熔断/降级/关闭 | 系统自动化 |
| L2 人工 | 15分钟-2小时 | 稳定核心功能 | 值班人员 |
| L3 修复 | 2-24小时 | 根因分析+修复 | 核心团队 |
| L4 复盘 | 24小时内 | 复盘+更新预案 | 事后团队 |

### L1自动响应清单

```
触发条件（任一）：
□ 错误率>10%
□ 响应时间>10秒
□ 成本>$5/次
□ 数据异常（偏离历史>3σ）

自动动作：
1. 熔断：停止接收新请求
2. 降级：返回缓存/简化结果
3. 告警：通知值班人员
4. 记录：保存现场日志
```

### L2人工接管清单

```
□ 确认故障范围（部分/全部）
□ 启动备用方案（如有）
□ 通知利益相关方（预估恢复时间）
□ 决定是否升级到L3
```

### L3修复清单

```
□ 根因分析报告
□ 修复方案及测试计划
□ 预计恢复时间
□ 修复后验证
```

### L4复盘模板

```
事件时间线：
├── 触发时间：
├── 发现时间：
├── L1启动时间：
├── L2接管时间：
├── 恢复时间：
└── 总耗时：

问题分析：
├── 根因：
├── 为何L1未自动触发（如适用）：
└── 为何L2延迟（如适用）：

改进措施：
├── 新增监控指标：
├── 新增自动响应规则：
├── 新增应急预案条目：
└── 责任人+完成时间：
```

### 禁止行为

```
□ 分析>15分钟才行动
□ L1未自动触发
□ L2延迟>30分钟
□ 无复盘或复盘延迟>24小时
```

### 度量指标

```
L1触发率 = 故障中L1自动触发的比例
目标：>95%

L2接管及时率 = 15分钟内人工接管的比例
目标：>90%

复盘完成率 = 24小时内完成复盘的比例
目标：100%
```

## 策略6：分层自治

### 分层定义

| 层 | 时间尺度 | 核心产出 | 不干预下层 |
|----|----------|----------|------------|
| 战略 | 月级 | 目标+边界+契约 | 具体实现细节 |
| 战术 | 周级 | 方案+接口定义 | 代码级实现 |
| 反应 | 秒/分级 | 执行结果 | 方案选择 |

### 层间契约格式

```
上游契约输入：
├── 目标描述：
├── 硬性约束：
├── 软性偏好：
└── 验收标准：

下游契约输出：
├── 方案概述：
├── 接口定义：
├── 依赖清单：
└── 风险说明：
```

### 契约变更流程

```
变更需求 → 触发Complicated域流程 → 72小时冷却
    ↓                                      ↓
变更批准 → 更新契约版本 → 通知下游       变更拒绝 → 维持原契约
    ↓
下游按新契约实现
```

### 禁止行为

```
□ 战略层直接修改代码
□ 反应层质疑战略方向
□ 层间无契约实时协调
□ 契约变更无冷却期
```

## 策略7：重心分配

### Type判定（决策前30秒）

```
Type 1：计算型
├── 特征：有明确最优解
├── 人类：设定目标
├── AI：执行
└── 资源：默认流程，快速通过

Type 2：分析型
├── 特征：多方案权衡
├── 人类：选择框架
├── AI：生成选项
└── 资源：分析时间，72小时冷却

Type 3：战略型
├── 特征：定义问题本身
├── 人类：主导定义
├── AI：挑战假设
└── 资源：80%投入，深度研讨
```

### 三问筛选法

```
判定问题是否为Type 3：

问1：此决策现在不定，系统会立即停滞？
├── 是 → 可能是Type 3
└── 否 → 继续

问2：此决策错误，6月后发现，代价不可承受？
├── 是 → 可能是Type 3
└── 否 → 继续

问3：此决策正确，能带来10倍收益？
├── 是 → 可能是Type 3
└── 否 → Type 1或2

结论：
任一问"是" → Type 3 → 80%资源
全部"否" → Type 1或2 → 20%资源
```

### 资源分配公式

```
总资源 = R

Type 3决策数量 = N3
Type 1/2决策数量 = N12

每个Type 3分配 = 0.8 × R / N3
每个Type 1/2分配 = 0.2 × R / N12

注：若N3=0，则20%资源作为储备；若N3>R，则按优先级排序
```

### 禁止行为

```
□ Type 3决策降级处理
□ Type 1决策占用Type 3资源
□ 未判定类型直接决策
□ 资源均匀分配
```

## 策略8：人机分工

### 分工矩阵

| 域 | Type | 人类角色 | AI角色 | 禁止行为 |
|----|------|----------|--------|----------|
| Clear | 1 | 设计契约 | 执行契约 | 人类干预执行 |
| Complicated | 2 | 选择框架 | 生成选项 | AI做最终决策 |
| Complex | 3 | 主导定义 | 挑战假设 | AI定义问题 |
| Chaotic | - | 接管L2-L3 | 执行L1 | AI接管L2-L3 |

### 人类责任清单

```
设计契约：
□ 输入格式定义
□ 输出格式定义
□ 错误处理方式
□ 性能基准

选择框架：
□ 明确决策目标
□ 定义评估维度
□ 设定约束条件
□ 做出最终选择

主导定义：
□ 问题边界
□ 核心假设
□ 成功标准
□ 价值判断
```

### AI责任清单

```
执行契约：
□ 按定义处理输入
□ 按格式生成输出
□ 异常时显式报错
□ 记录执行日志

生成选项：
□ 至少3个可行方案
□ 各方案优缺点分析
□ 风险评估
□ 推荐排序（供参考）

挑战假设：
□ 列出隐含假设
□ 提供反对观点
□ 模拟极端场景
□ 指出逻辑漏洞
```

## 策略9：冲突解决

### 优先级（降序）

```
1. 域判定规则
2. Chaotic域覆盖（危机时）
3. 分层原则
4. 重心分配
5. 人机分工
6. 域内具体规则
```

### 冲突处理流程

```
识别冲突 → 识别冲突的规则层级
    ↓
层级明确？ → 按优先级解决
    ↓
层级不明确？ → 升级处理 → 人工判定
    ↓
记录冲突 → 复盘时分析 → 更新规则
```

### 默认策略

```
域不明确 → 默认Complex（最保守）
类型不明确 → 强制Type 3（最保险）
规则不明确 → 暂停决策，人工升级
```
