# 因果层 - 因果链提取指令设计调研

## 核心问题

如何设计 Prompt 让多视角 Agent 从文档中高效、一致地提取因果链（CausalLink），保证输出格式标准化以便后续融合。

---

## 备选方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **A: Zero-shot + 详细格式说明** | 简单直接，无需示例 | 输出一致性差，格式易偏离 | 快速原型验证 |
| **B: Few-shot + 3-5个示例** | 输出稳定，格式一致 | 示例设计耗时，可能过拟合 | **推荐主方案** |
| **C: Structured Output (JSON Schema)** | 强制格式，无解析错误 | 限制 LLM 灵活性，字段僵化 | 生产环境确定性需求 |
| **D: Function Calling / Tool Use** | 最严格的结构化输出 | 复杂度高，依赖特定模型能力 | 复杂多步骤提取 |

## 学术/工业界实践

### 1. Agentic Leash (arXiv 2601.00097, 2026)

**核心发现**：
- Single-Agent 即可完成 CLD 提取，无需多 Agent
- **关键**：Prompt 中明确指定"系统思考者"角色，要求识别变量和因果连接
- 输出格式：JSON 列表，每项含 `from`, `to`, `polarity`

**Prompt 模板**（简化版）：
```
你是一位系统思考专家。阅读以下文本，识别所有因果变量和它们之间的连接。

对于每个因果连接，输出：
- from: 原因变量（具体名词短语）
- to: 结果变量（具体名词短语）
- polarity: "+"（促进）或 "-"（抑制）

只输出 JSON 数组，不要有其他解释。

示例：
[
  {"from": "政府补贴", "to": "房价", "polarity": "+"},
  {"from": "利率上升", "to": "购房需求", "polarity": "-"}
]
```

### 2. GraphRAG (Microsoft, 2024)

**实体-关系提取方法**：
- 使用 LLM 提取实体（节点）和关系（边）
- **分两步**：先提取实体，再识别关系
- 输出格式：GraphML 或自定义 JSON

**关键洞察**：
- 分步提取比一步提取准确率更高（实体识别错误不影响关系判断）
- 需要实体消歧（同一概念不同表述）

### 3. PersonaFlow (arXiv 2409.12538)

**动态视角方法**：
- 不同专家角色关注不同变量类型
- 政策专家关注"法规/政策"类变量
- 经济专家关注"市场/价格"类变量

**实现方式**：
- System Prompt 定义角色和关注焦点
- Task Prompt 定义提取任务和格式

### 4. 领域实践：CausalNLP

**因果抽取主流方法**：

| 方法 | 代表工作 | 特点 |
|------|----------|------|
| 基于规则 | OpenCog Pattern Matcher | 高精度，低召回，维护成本高 |
| 基于分类器 | Causal-BERT | 需要标注数据，领域迁移差 |
| 基于 LLM | GPT-4 + Prompt Engineering | 通用性强，成本低，一致性需优化 |
| 混合方法 | Rule + LLM | 规则保底，LLM 扩展 |

## 推荐方案

### 分两步提取（推荐）

**Step 1: 实体识别**（各视角 Agent 并行）
```
你是{perspective}专家。从以下文档中识别与{domain}相关的所有关键变量。

要求：
1. 变量必须是具体名词短语（2-5个词）
2. 排除抽象概念如"影响"、"结果"
3. 优先选择可量化或明确可观测的变量

输出格式：每行一个变量，格式 "- 变量名: 简要说明"

示例：
- 政府住房补贴: 政府向购房者提供的财政支持
- 房贷利率: 银行向购房者收取的贷款利息
```

**Step 2: 因果连接识别**
```
基于已识别的变量列表，阅读文档找出它们之间的因果关系。

对于每对具有因果关系的变量，输出：
```
{from} → {to} | {polarity} | {evidence}
```

规则：
- polarity: "+" 表示增加/促进，"-" 表示减少/抑制
- evidence: 原文引用，标注页码

示例：
政府住房补贴 → 房价 | + | "政府补贴增加了购房需求，推动房价上涨 (p.12)"
```

### 输出格式标准化

**CausalLink 数据结构**：
```json
{
  "source": "政府住房补贴",
  "target": "房价",
  "polarity": "+",
  "evidence": "政府补贴增加了购房需求，推动房价上涨 (p.12)",
  "source_agent": "policy-perspective",
  "confidence": 0.85,
  "extracted_at": "2026-04-13T10:30:00Z"
}
```

**字段说明**：
| 字段 | 必填 | 说明 |
|------|------|------|
| source | ✅ | 原因变量名（标准化后的） |
| target | ✅ | 结果变量名（标准化后的） |
| polarity | ✅ | "+" 或 "-" |
| evidence | ✅ | 原文引用+位置 |
| source_agent | ✅ | 提取该链接的 Agent ID |
| confidence | ⚪ | LLM 自评置信度（0-1） |
| extracted_at | ⚪ | 时间戳 |

## 待决策事项

1. **分步 vs 一步提取**
   - 选项 A：一步完成（实体+关系同时提取）—— 简单但可能遗漏
   - 选项 B：分两步（先实体后关系）—— 复杂但更可靠
   - **建议**：Phase 1 用一步，Phase 2 优化为两步

2. **置信度字段**
   - 选项 A：要求 LLM 自评（增加 Prompt 复杂度）
   - 选项 B：不收集，后续通过冲突检测推断
   - **建议**：Phase 1 不收集，减少复杂度

3. **Few-shot 示例数量**
   - 选项 A：1-2 个简单示例
   - 选项 B：3-5 个覆盖不同场景
   - **建议**：3 个示例（正例、负例、复杂例各一）

4. **变量标准化时机**
   - 选项 A：提取时立即标准化（Agent 内部）
   - 选项 B：提取后统一标准化（Conductor 层）
   - **建议**：B，避免 Agent 间标准不一致
