# 研究型 Agent MVP 现状摸底报告

## 1. 现有系统能做什么、不能做什么

### 现有系统已经能做什么

- 已有统一业务入口 `RAGService`，可以承接索引构建、单轮查询和多轮聊天，并把结果统一包装成 `RAGResponse` / `ChatResponse`。
- 已有两条查询执行路径：
  - `ModularQueryEngine` 负责查询处理、检索路由、后处理、答案生成和引用格式化。
  - `AgenticQueryEngine` 负责用规划 Agent 在一次查询内自主选择检索策略，并带有基础的超时与最大迭代控制。
- 已有最小会话能力：
  - `ChatSession` / `ChatManager` 可以维护聊天历史、生成会话标题，并在一次对话内复用上下文。
  - 这套状态是“聊天状态”，不是“研究状态”。
- 已有标准查询出入口 `backend/business/rag_api/rag_service_query.py`，它把底层 `query_engine.query()` 的结果转成带 `sources` 和 `metadata` 的 `RAGResponse`。
- 已有引用来源模型和推理链字段，说明系统已经具备“带证据返回结果”的基础外壳。

### 现有系统还不能做什么

- 还没有“研究内核”这一层。当前系统能回答一次问题，但不能围绕研究问题持续推进 `问题 -> 取证 -> 判断 -> 收束`。
- `AgenticQueryEngine` 仍是“一次查询内的检索策略自主”，不是“跨轮研究任务自主”。它没有维护研究问题边界、阶段性判断、关键张力、候选下一步动作。
- 会话层只记录聊天问答历史，不记录研究过程所需的状态，例如证据账本、当前判断、未解决冲突、预算消耗和停止原因。
- 当前输出主形态仍是“答案 + 来源”，而不是“判断 + 证据依据 + 不确定性 + 下一步追问”。这与北极星文档要求的研究型交付还有距离。
- 已知问题已经暴露出一些研究链路无法直接继承的风险：
  - Agentic sources / reasoning 提取仍是简化实现，证据回溯不够稳。
  - `max_llm_calls` 只定义未落地，成本控制不完整。
  - 流式能力是降级实现，不影响 MVP 核心，但说明 agentic 这条链路还不适合继续扩张职责。

## 2. 研究内核缺失的具体能力清单

1. **问题定焦与子查询规划能力**
   当前只有查询改写，没有面向研究任务的“把大问题拆成 2-3 个可验证子问题”的能力。

2. **研究状态承载能力**
   当前只有聊天历史，没有研究问题边界、证据账本、阶段性判断、未决问题和停止原因这些状态。

3. **判断形成能力**
   当前系统擅长生成答案，但没有一个明确合约去区分“摘要”与“判断”，也没有显式产出 `confidence`、`next_questions`。

4. **预算内主动收束能力**
   当前最多是单次查询里的超时/迭代限制，没有围绕 `budget_turns` 做“继续取证 / 收束 / 停止”的决策。

5. **稳定的证据回溯能力**
   当前虽然能返回 `sources`，但研究型输出需要可回溯的 `evidence_refs`，而现有 agentic 路径的来源提取本身仍有已知不确定性。

## 3. 本次 MVP 的 Scope 边界决定

### 决定 1：只做单 Agent、单次运行内的研究闭环，不接多 Agent

**理由**：北极星文档已经明确第一版不预支多 Agent 复杂度。当前仓库也没有研究任务编排器，多 Agent 会直接把范围推爆。

### 决定 2：研究内核放在 `backend/business/research_kernel/`，作为业务层编排模块

**理由**：研究内核的职责是“把证据推进成判断”，属于业务逻辑，不应该落到基础设施层，也不应直接塞进前端或现有聊天会话层。

### 决定 3：Phase 3 只复用现有 `rag_service_query.execute_query()` 查询路径，不重写检索

**理由**：本轮目标是验证研究主链路，不是替换底层 RAG。复用现有查询路径可以保持接口兼容，并降低对现有引擎的侵入。

### 决定 4：MVP 只维护“本次运行内的最小研究状态”，不接持久化研究记忆

**理由**：现有 `ChatSession` 是聊天历史模型，不适合直接承载研究账本。短周期 MVP 先用运行内状态完成闭环，避免误把持久化设计做成主任务。

### 决定 5：MVP 输出聚焦 `judgment + evidence_refs + confidence + next_questions`，不扩展 UI、模板系统或报告协议

**理由**：这能直接验证“判断区别于摘要”是否成立，也符合 8 小时预算内的最小可运行目标。报告编排、启发式展示和复杂前端交互应后置到下一阶段。
