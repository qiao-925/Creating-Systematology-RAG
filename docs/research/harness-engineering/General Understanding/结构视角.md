# 结构视角：先给结论

如果只记一句话：**从结构上看，Agent 不是“会调用工具的模型”，而是“围绕目标、状态、行动和反馈组织起来的闭环系统”。**

这句话听起来抽象，但它能帮你把很多混乱的讨论一下子分开：

1. 有些系统只是把 LLM 嵌进了既定流程里，那更接近 workflow。
2. 有些系统真的把“下一步做什么”的部分决策权交给了模型，那才开始进入 agentic 结构。
3. 有些系统虽然也会调工具，但没有稳定的状态管理、退出条件、权限边界和观察能力，本质上还是“带插件的聊天”。

所以，所谓“结构视角”，不是先问有几层，而是先问三件事：

1. **谁决定下一步？**
2. **系统如何接触外部世界？**
3. **行动结果如何被保留、修正和约束？**

后面所有的层，几乎都在回答这三个问题。

# 一、先把边界摆正：什么才算 Agent

Anthropic 在 S3 里给了一个非常重要的分界：**workflow 是预先写好的代码路径在编排 LLM 和工具；agent 则是模型动态决定过程和工具使用方式。** 这不是概念洁癖，而是结构上的分水岭。

为什么这条边界这么重要？因为它决定了系统复杂度会长在什么地方。

- 如果系统是 workflow，复杂度主要长在代码分支、状态机、规则树和异常路径里。
- 如果系统是 agent，复杂度开始长在模型决策、工具选择、状态压缩、反馈循环和治理边界里。

换句话说，**结构上真正要盯住的不是“有没有工具”，而是“决策权放在哪里”。**

这也是很多初学者最容易误判的地方。一个“调用搜索接口然后总结结果”的脚本，不会因为接了工具就自动升级成 Agent；反过来，一个能根据观察结果改写计划、决定继续查证还是停止、知道何时请求人类确认的系统，即便只接了很少的工具，也更接近 Agent。

Anthropic 进一步提出了“augmented LLM”的概念：LLM 加上检索、工具、记忆之后，才是 agentic system 的基础构件。注意，是**基础构件**，不是完整答案。因为“增强后的模型”只解释了它拥有什么能力，还没解释这些能力如何被组织成一个稳定闭环。

note：“Agent和Workflow的核心区别 —— 决策权”

# 二、一张比“分层罗列”更有用的结构图

旧文档《通用Agent架构.md》给出的价值在于提醒我们：Agent 绝不是单层软件，它至少跨过接入、应用、规划、执行、记忆、模型、能力和基础设施这些面向。但如果只停在“分层图”，读者很容易看完以后仍然不知道：**这些层到底为什么存在，它们之间真正的控制关系是什么。**

从学习视角，我更建议先记住下面这张图：

```text
用户目标 / 外部事件
          |
          v
决策内核：模型 + 指令
          |
          v
运行循环：理解 -> 计划 -> 行动 -> 观察 -> 决定继续/停止
          |
          v
状态系统：会话、短期记忆、中间结果、长期记忆
          |
          v
能力接口：函数、检索、数据库、网页、文件、MCP 工具
          |
          v
外部世界：业务系统、知识源、执行环境、人

外圈持续包裹：
- 护栏与权限
- 日志、Tracing、评测
- 人类介入
- 协议与互操作
- 多 Agent 编排（当单 Agent 不够时）
```

这张图比传统分层图更有解释力，因为它把 Agent 的结构核心压缩成两条主线：

- 一条是**控制线**：系统怎么决定下一步、怎么循环、怎么停。
- 一条是**信息线**：系统拿到了什么观察、保留了什么状态、把什么再送回模型。

只要这两条线看清楚，很多架构术语都会自动落位。

# 三、核心结构：真正不可再删的四个部分

下面这四个部分，我认为是 Agent 结构上真正的“骨架”。删掉任何一个，系统都很容易退化成聊天、脚本或者脆弱自动化。

## 1. 决策内核：模型 + 指令

OpenAI 在 S1 里把 Agent 最基本的三个部件写得很清楚：**模型、工具、指令**。其中最先要理解的是“模型 + 指令”为什么必须被放在一起看。

模型提供的是泛化推理、语言理解和不确定环境下的选择能力；指令提供的是目标边界、行为约束和优先级。没有模型，你只有静态规则；没有指令，你得到的是一个能力很强但目标感很弱的概率机器。

所以，结构上所谓“决策内核”，并不是一个抽象的 LLM API，而是：

- 这个系统准备用哪个模型承担哪类判断；
- 这些判断受到什么指令、政策和角色边界约束；
- 当多个目标冲突时，优先级如何被表达给模型。

从这里也能看出一个关键洞察：**Prompt 不是整个架构，但它确实属于架构。** 因为它直接决定了“决策内核”如何行使决策权。

## 2. 行动接口：工具不是附属插件，而是通向外部世界的结构面

OpenAI 在 S1 中把工具定义为 Agent 采取行动的外部函数或 API；Google Cloud 在 S4 中又补了一层：工具不仅是“能力扩展”，还是把 Agent 接到真实数据和真实系统上的结构接口。

这意味着工具层的意义远不止“多一个功能”。

它至少承担三件事：

1. 让模型接触训练数据之外的实时信息。
2. 让模型不仅能说，还能查、算、写、改、执行。
3. 让“语言层面的意图”被翻译成“系统层面的动作”。

因此，工具层不是装饰，而是 Agent 连接环境的接口面。没有它，模型只能在文本里假装完成任务；有了它，Agent 才真正开始影响世界。

但这也带来一个反直觉事实：**工具越多，不代表 Agent 越强。** 微软在 S9 中讨论的 tool-space interference，恰好说明当工具空间过大、命名冲突、返回过长、功能重叠时，整体效果反而会下降。也就是说，工具层首先是“可选择的行动空间”，其次才是“功能清单”。

note:"关于工具，可以使用agentskill的渐进式披露设计吗？还是已经实现了目前？"

补充回答：可以，而且我会把它理解成“工具暴露面的渐进式披露”，但不能把 skill 和 tool 混成一层。skill 管的是“何时用、如何用、有哪些规则和步骤 relevant”；tool 管的是“能做什么、输入输出是什么、权限边界在哪”。把 Agent Skills 的渐进式披露思想迁到工具层，最合适的做法不是把所有工具一次性塞给模型，而是先暴露粗粒度能力，再按需要下钻到更细的工具集合。OpenAI Agents SDK 里已经能看到这条思路的实现，例如 `tool namespaces`、`ToolSearchTool`，以及 `HostedMCPTool` 的 `defer_loading`；Anthropic 的 skill 设计则强调 frontmatter 常驻、正文按需加载，本质上也是在控制“当前任务真正需要看到多少能力说明”。所以更准确的结论是：**这个设计思想完全可以复用，而且主流系统已经部分实现，但大多只实现到“能力按需暴露”这一层，还没有天然替你完成“使用策略按需教学”这一层。** 后者仍然要靠 prompt、skill、examples 和 eval 一起补上。

## 3. 运行循环：Agent 的中心不是模型本身，而是 loop

很多人理解 Agent 时，最容易漏掉的就是这一层。OpenAI 在 S1、S2 中都强调了 `agent loop`：系统会在工具调用、结果回传、继续决策之间反复循环，直到达到退出条件。Google Research 的 ReAct（S5、S6）则给出了更强的理论说明：推理和行动不是两段式流程，而是应当交替出现，形成“reason to act / act to reason”的闭环。

这件事的结构意义非常大。

如果没有 loop，系统就只是一次生成。
如果有了 loop，系统才开始具备下面这些能力：

- 把一个大目标拆成多步；
- 根据观察结果修正原计划；
- 在错误和意外出现时继续尝试，而不是一次失败就结束；
- 通过退出条件控制何时停止，而不是无穷调用。

所以，**Agent 和普通聊天系统最深的差别，不在于会不会调工具，而在于有没有“持续到完成或终止”的运行循环。**

note：“目前这块的最佳实践有哪些？如何判断要不要继续尝试？又是何时退出？这块目前是怎么设计的？”

补充回答：截至 2026 年 3 月，比较稳定的实践不是“让 Agent 一直试”，而是给 loop 明确的预算和退出语义。常见做法有：给循环设置 `max_turns`、工具调用预算和时间预算；区分可重试错误与不可重试错误；在每轮后判断“再试一次是否会带来新信息”；高风险动作前插入 human approval；把**成功完成、受控失败、升级给人**三类结束状态都做成**显式分支**。判断要不要继续尝试，核心看四件事：有没有新观察、失败是否可恢复、替代路径是否存在、预算是否还值得烧。连续两三次命中同一类错误、缺权限/缺前置条件、或下一步只是在重复同一假设时，就该退出或升级，而不是继续盲试。OpenAI Agents SDK 当前的运行设计基本也是这个思路：工具调用 -> 执行 -> 把结果送回模型 -> 再决策；当模型不再调用工具、产出最终输出、命中 `final_output`、发生错误、或超过 `max_turns` 时停止。换句话说，**最佳实践不是“多试”，而是“有证据地试，有预算地停”。**

note+：“这是否意味着agent loop当前的实现大部分仍是由人来控制，如限制max turns, 工具，时间预算等显式设计，而非交由agent决策。”

是，基本可以这样理解；但更准确的表述是：

  当前主流的 agent loop，大多不是“把循环控制权完全交给 agent”，而是“由开发者/宿主系统定义边界与治理规则，agent 在这些边界内做局部决策”。

  截至 2026 年 3 月 17 日，从 OpenAI Agents SDK、Google ADK、LangGraph 这些主流框架的官方文档看，整体都是这个模式。这是我基于这些官方文档做出的归纳，而不是单看某
  一家实现的猜测。

  更精确地拆开看，有三层控制权：

  - 外层控制权通常在人和宿主程序手里：max_turns / maxTurns、max_iterations、recursion_limit、工具白名单、审批点、guardrails、异常处理、暂停/恢复、状态持久化。
  - 内层策略权通常在 agent/模型手里：这一轮要不要调工具、调哪个工具、是否 handoff、根据 observation 要不要改计划、什么时候给出 final output。
  - 硬停止权几乎总在框架或宿主侧：预算耗尽、审批被拒、guardrail 命中、权限缺失、超时、不可恢复错误。

  所以，不是“人控制每一步”，而是：

  人控制制度和边界，agent 控制边界内的下一步动作。

  拿 OpenAI 官方实现来说，这个边界非常明确：

  - OpenAI Agents SDK 的 Runner loop 是固定语义：调用模型；如果产出 final_output 就返回；如果 handoff 就切到新 agent 继续；如果有工具调用就执行工具并把结果送回模
    型继续；超过 max_turns 就抛 MaxTurnsExceeded 或进入你定义的错误处理。OpenAI Python Running Agents
    (https://openai.github.io/openai-agents-python/running_agents/) / OpenAI JS Running Agents (https://openai.github.io/openai-agents-js/guides/running-agents/)
  - 可用工具是开发者预先配置给 Agent(..., tools=[...]) 的，不是 agent 自己无限发现和无限扩张权限。OpenAI Tools
    (https://openai.github.io/openai-agents-python/tools/)
  - 高风险工具是否需要审批，也是开发者通过 needs_approval、require_approval 这类机制显式定义；命中后运行会暂停，等待人批准或拒绝，再恢复。OpenAI Human-in-the-loop
    (https://openai.github.io/openai-agents-python/human_in_the_loop/)
  - guardrails 命中 tripwire 时，SDK 会直接中止执行，而不是让 agent 自己“再想想”。OpenAI Guardrails (https://openai.github.io/openai-agents-python/guardrails/)

  更关键的是，这不只是 OpenAI 一家的风格。其他主流框架也在做同样的事：

  - Google ADK 的 LoopAgent 文档直接写得很清楚：它本身并不会天然决定何时停止循环，你必须提供终止机制，比如 max_iterations 或一个子代理发出的停止信号。Google ADK
    Loop Agents (https://google.github.io/adk-docs/agents/workflow-agents/loop-agents/)
  - LangGraph 明确提供 recursion_limit，达到上限会抛 GraphRecursionError；也提供 human-in-the-loop interrupt，让工具调用在命中策略时暂停，等待人类决策后再继续。L
    angGraph Graph API (https://docs.langchain.com/oss/python/langgraph/graph-api) / LangChain Human-in-the-loop
    (https://docs.langchain.com/oss/python/langchain/human-in-the-loop)

  所以，你这句 note+ 可以进一步打磨成下面这个版本，更严谨：

  是。这意味着当前 agent loop 的主流实现，本质上仍是“受治理的自治”而不是“完全自治”。开发者负责定义循环的预算、工具边界、审批机制、异常分流和停止条件；agent 负责在
  这些边界内根据当前观察决定下一步。也就是说，是否继续、能做什么、何时必须停，今天大多仍由人和宿主系统通过显式机制兜底，而不是完全交给 agent 自我裁决。

## 4. 状态系统：上下文不是记忆，记忆也不只是向量库

Google Cloud 在 S4 中对这件事说得很清楚：短期记忆至少涉及 session 和 state，长期记忆则涉及跨会话的持久知识。OpenAI SDK 在 S2 里把 sessions 明确列为工作上下文的持久层。这两个来源合起来，足够纠正一个很常见的误区：**把上下文窗口误当成记忆。**

从结构上讲，状态系统至少包含三类东西：

1. **会话状态**：当前任务进行到了哪一步，用户和 Agent 交换了什么，中间工具结果是什么。
2. **工作记忆**：为了完成当前任务必须保留的中间结论、待办项、草稿和错误反馈。
3. **长期记忆**：跨任务、跨会话仍然需要保留的用户偏好、历史事实、可复用知识。

为什么这层必须独立？因为模型上下文窗口只是“本轮可读到的文本”，它既不天然持久，也不天然结构化，更不天然知道什么该保留、什么该忘记。真正的状态系统要解决的是：**哪些信息进入模型，哪些信息留在外部存储，哪些信息要被压缩、摘要、检索和淘汰。**

没有这层，Agent 做长任务时会很快退化成“每一轮都像第一次见面”。

# 四、扩展结构：当 Agent 进入真实环境后，外圈会长出来什么

上面四个部分解决的是“能不能算一个 Agent”。但一旦系统进入真实业务环境，结构会继续向外生长。下面这些部分，我更愿意称为**扩展结构**。它们不是概念上的起点，却往往是生产可用性的关键。

## 1. 多 Agent 编排

OpenAI 在 S1 中把多 Agent 看成单 Agent 之外的编排模式，Anthropic 在 S3 中也一再提醒“先从简单、可组合模式开始”。这背后的结构判断非常重要：**多 Agent 不是更高级的默认答案，而是当职责边界已经明显到单 Agent 难以承载时，才有意义的扩展。**

微软的 Magentic-One（S8）展示了一个典型模式：由 Orchestrator 协调多个专长不同的 Agent。它适合什么情况？适合这些能力之间已经形成比较稳定的职责分工，例如网页浏览、终端执行、文件处理、代码编写分别由不同子 Agent 负责。

什么时候不该急着上多 Agent？当你只是觉得“复杂任务听起来应该多智能体”时。因为结构一旦裂成多个 Agent，你不仅增加能力，也增加了：

- 协调成本；
- 上下文同步成本；
- 失败定位难度；
- 权限边界设计难度。

note：“这块可以举例说明一下吗？如codex的spawn agent设计和claude code review的多agent设计，结合例子说明设计哲学”

补充回答：这块很适合拿两个例子对照。

以当前 Codex 的 `spawn_agent` 设计看，它的哲学更像“主线程保持控制，侧任务并行外包”：阻塞当前下一步的关键工作，优先主 Agent 自己做；能够并行、边界清晰、且不会和主线争抢上下文的工作，才交给子 Agent。比如主 Agent 在修一个认证问题时，可以把“快速摸清模块结构”“并行跑另一组测试”交给 explorer 或 worker，但不会把“判断根因是什么”这种关键路径决策先丢出去等待结果。这种设计优先的是**吞吐量、上下文聚焦和可集成性**。

Claude 侧则更明显地走“角色隔离和上下文隔离”路线：一类是 subagent，在单会话内承担专门职责、拥有独立 context window 和受限工具；另一类是更强的 team/code review 设计，Anthropic 公开文档里明确写到 code review 会用一组 `specialized agents` 并行分析，再经过 `verification step` 过滤误报，最后才汇总为 review 结论。这说明它追求的不只是并行本身，而是**多视角发现问题 + 独立验证降噪**。Anthropic 在 2025 年 6 月 13 日讨论多 Agent 的研究文章里也明确指出，**多 Agent 最适合那些可以天然并行、方向彼此相对独立的探索任务；而很多编码任务其实依赖耦合很强，并不适合盲目并行。**所以可以把两者的设计哲学概括成一句话：**Codex 的 `spawn_agent` 更像“经理-工人”式工程调度；Claude 的 review 体系更像“多位专职审稿人并行发现，再交给验证环节收敛”。**

## 2. 协议与互操作层

MCP 官方规范（S7）和 Google Cloud 架构文档（S4）共同说明了一个关键点：**协议层的作用，是把“如何连接工具”标准化，而不是替你解决所有治理问题。**

MCP 为什么重要？因为当 Agent 不再只接自家手写函数，而是要接第三方能力、共享 server、跨宿主环境运行时，协议层就成为一个独立结构问题。它要回答的是：

- host、client、server 谁负责什么；
- 会话怎样建立；
- 能力怎样协商；
- 不同工具之间如何隔离；
- 上下文由谁汇总，谁不该看到全部对话。

这层一旦被看见，你就会意识到：**工具本身是一回事，工具接入协议是另一回事。**

## 3. 治理与安全层

OpenAI 在 S1、S2 中都把 guardrails 放在显著位置，同时明确指出护栏要和认证、授权、访问控制一起工作。换句话说，护栏不是万灵药，它只是治理层的一部分。

结构上，这一层主要处理的是“模型有了行动能力以后，什么不该做、什么必须确认、什么超出权限、什么需要中断”。

这里真正重要的不是把安全做成一个单点，而是像 OpenAI 所说的那样做成**分层防线**：

- 输入侧检查；
- 输出侧检查；
- 工具调用前后的约束；
- 高风险动作的人类确认；
- 与认证授权、访问控制联动。

你可以把它理解成：核心结构负责“让 Agent 能做事”，治理层负责“让 Agent 做事时不越界”。

## 4. 观测、Tracing 与评测层

很多架构图喜欢把日志、监控、Tracing 画在最底下，像基础设施附属品。但 OpenAI SDK 在 S2 里把 Tracing、evaluation、fine-tuning 支持直接和 Agent 原语并列出现，这其实是在提醒我们：**对于 Agent，这些不是可有可无的运维细节，而是理解其行为的必要结构。**

原因很简单。Agent 不是一步产出，而是多步决策。只看最终答案，你往往根本不知道问题出在：

- 指令不清；
- 工具选错；
- 参数传错；
- 观察结果被误读；
- 状态污染；
- 循环退出条件设计不当。

所以，传统软件里的“可观测性”到了 Agent 这里，不只是排障能力，更是学习和迭代 Agent 的主入口。

note：“可观测性从另一个角度来说是对模型幻觉的补充？”

补充回答：可以这样理解，但要更精确一点。可观测性不是“抑制幻觉的另一种模型能力”，而是**让幻觉和其他错误可被识别、定位、复盘**的结构层。幻觉只是一类失败；在 Agent 系统里，错结果也可能来自工具选错、参数传错、检索 miss、状态污染、handoff 错位，或者 stop condition 设计失灵。没有 trace，你只看到“答案错了”；有 trace，你才能继续追问：它到底是无中生有，还是基于错误观察做了看似合理的推断。所以它确实是对抗幻觉的重要补充，但更准确地说，它补的是**整体错误归因能力**。防幻觉本身还得靠检索、校验、引用约束、工具验证和 guardrails；observability 负责让这些机制能被调优，而不是替代它们。

可观测性 -> 增强整体归因能力！

## 5. Human-in-the-loop

OpenAI SDK 在 S2 里把 Human-in-the-loop 直接列成内建能力；Anthropic 在 S3 里也强调高价值 Agent 往往伴随有意义的人类监督。结构上，这说明“人工介入”不只是运营策略，也是一层正经的控制面。

它典型出现在三类场景：

1. 高风险动作执行前确认。
2. 目标不明确、需要价值判断时升级给人。
3. 失败多次、超出策略边界时请求人工接管。

如果你把这层拿掉，系统就会被迫在“不该自动化”的地方也继续自动化。

note：“这段很适合沉淀为skill， 人类的价值显现于此， 风险控制， 价值判断， 转人工”

补充回答：同意，这一段非常适合沉淀成 skill，不过最好拆成“组织规则 + 运行时机制”两层。作为 skill，它可以规定哪些场景必须转人工、触发信号是什么、转交时需要打包哪些上下文、人工回来后如何恢复；作为运行时机制，它则对应审批门、暂停/恢复、权限升级、人工接管等控制点。人类在这里的价值不是“替模型补一句话”，而是承担模型不该擅自承担的三类责任：价值判断、风险签字、异常兜底。也因此，**转人工不是系统失败，而是把责任边界显式化。**

note+：“这块目前各个厂商是如何设计的？例如在codex cli和claude code cli中， 研究一下”

补充回答：截至 2026 年 3 月 17 日，我核对的公开官方资料里，这一层已经不是抽象原则，而是被厂商明确做成了产品控制面，不过实现重心并不一样。`Codex CLI` 目前公开得最清楚的是“沙箱 + 审批策略”这两层：OpenAI 直接把 `sandbox_mode` 和 `approval_policy` 做成一等配置，前者限制 agent 能碰到的执行环境，后者决定什么时候必须把控制权交还给人。公开文档里，审批策略至少区分 `untrusted`、`on-failure`、`on-request`、`never`，沙箱至少区分 `read-only`、`workspace-write`、`danger-full-access`；初始模式还会参考当前目录是否受版本控制、是否已被标记为 trusted。换句话说，Codex 的设计重点是：**先根据环境风险设默认边界，再在执行时通过审批门做人工介入。** 这非常适合“高风险动作执行前确认”和“失败后升级处理”两类场景，也说明 Human-in-the-loop 在它这里首先是一套运行时权限控制机制。

`Claude Code` 则把这层公开得更细，更接近“组织规则 + 运行时事件”分离。Anthropic 官方文档里至少能看到四组机制：一是 `permission mode`，把常见交互拆成 `default`、`acceptEdits`、`plan`、`bypassPermissions` 等模式；二是 `allow/deny rules`，允许长期放行或长期拦截某些命令与工具；三是 `hooks`，可以在 `PreToolUse`、`PostToolUse`、`Notification`、`UserPromptSubmit`、`Stop`、`SubagentStop` 等节点插入外部检查、阻断或记录逻辑；四是 Agent SDK 里的 `PermissionRequest` 与 `Elicitation`，前者对应“这个动作要不要批”，后者对应“这里需要用户补信息或补价值判断”。这说明 Claude Code 的 Human-in-the-loop 不只是一道确认弹窗，而是被拆成了**权限控制、用户补充判断、流程中断与恢复、外部策略注入**几类不同控制点。

如果把两者放在一起看，一个很有价值的结论就出来了：主流厂商并不是把“转人工”做成单一功能，而是拆成两层。第一层是 policy layer，规定什么情况下必须让人介入；第二层是 runtime checkpoints，也就是审批、问询、hooks、停止、恢复、权限升级这些具体插桩点。`Codex CLI` 当前公开得更偏执行审批和环境边界，`Claude Code` 当前公开得更偏细粒度权限事件和可编排的中断点。至于“目标不明确、需要价值判断时升级给人”这一类，Claude Code 已经在 SDK 里把它公开成 `Elicitation` 这类单独事件；Codex CLI 的公开文档目前没有看到一个同等显式的独立原语，这一句更适合标注为**基于当前公开资料的判断**，不要写成已经确认的架构事实。

人工介入场景设计：高风险操作，价值判断，异常兜底。

# 五、为什么这些层不能塌成一团

理解 Agent 时，一个很有用的办法是问：**每一层到底在替系统消除哪一种失败模式？**

- 决策内核解决的是：面对模糊目标，系统如何做语义判断。
- 运行循环解决的是：一次生成不够时，系统如何持续推进任务。
- 状态系统解决的是：多轮、多步之后，系统如何不丢上下文。
- 行动接口解决的是：系统如何把文本判断转成外部动作。
- 协议层解决的是：这些动作接口如何标准化接入和互操作。
- 治理层解决的是：系统在能行动之后，如何不越界。
- 观测层解决的是：系统做错事之后，如何知道错在哪。

这就是为什么它们不能简单合并成“一个大 Prompt + 一堆工具”。

把它们揉在一起，短期看实现好像更快，长期看会出现三类问题：

1. **问题不可定位**：你不知道失败是模型问题、状态问题、工具问题还是协议问题。
2. **能力不可替换**：换一个工具、一个模型、一个存储，系统就牵一发而动全身。
3. **边界不可治理**：安全、权限、成本、延迟和质量会互相缠在一起，无法独立调节。

从这个意义上说，Agent 的“分层”本质上不是为了漂亮，而是为了让不同失败模式能被隔离、观测和替换。

note：“从prompt到agent，有点像是一种系统性的工程设计， prompt也能跑， 但agent架构更加稳定和可靠以及长期的可持续性更👍”

补充回答：这个理解是对的，而且可以再往前推一步：Prompt 解决的是“单次或浅层交互里，怎么把模型引导得更好”；Agent 架构解决的是“当任务跨时间、跨工具、跨状态后，怎么让系统持续可靠地完成事”。所以它不是 Prompt 的替代物，而是 Prompt 在真实环境里遇到稳定性、可追踪性、权限、记忆和恢复问题后，逐步外化出的工程结构。也正因如此，Prompt 仍然在架构内部，只是它不再单独扛全部复杂度。

# 六、常见的边界误判

## 1. 把 workflow 当 Agent

这是最常见也最根本的误判。只要下一步路径是开发者在代码里提前写死的，它就更接近 workflow。Agent 的关键不是用了 LLM，而是模型是否在真实执行中拥有一定的路径选择权。

## 2. 把 Prompt 当成全部架构

Prompt 很重要，因为它属于决策内核的一部分；但 Prompt 不是全部架构。它替代不了状态系统、工具接口、退出条件、授权控制和观察机制。只调 Prompt，却不改 loop 和 state，通常只会把问题推迟几轮再暴露出来。

note：“这块蛮有意思， 我这么理解， 本质上是需求， 首先用prompt来实现， 因prompt的局限性， ‘分化’成了上述的各种状态系统，工具接口等特定解决一类问题的方案； 刚刚还遇到一个codex的agents.md和skill的区别， 本质上是一件事， 但使用了不同程度的解决方法”

补充回答：你的理解基本成立。很多需求一开始确实都是“先用 prompt 试着跑通”，随后因为失败模式逐渐稳定，才把复杂度拆分到更专门的结构里：状态系统解决跨轮一致性，工具接口解决对外行动，guardrails 解决边界，observability 解决复盘，memory 解决长期积累。换句话说，**不是先有一套宏大架构，再去找问题；而是问题的不同失败模式，逼出了不同的结构件。** `AGENTS.md` 和 skill 也可以这样看：它们都属于“控制模型行为的上下文工件”，但粒度和生命周期不同。`AGENTS.md` 更像仓库级、长期常驻的行为宪法，回答“这个 Agent 在这个环境里总体该怎么协作”；skill 更像任务级、按需加载的作业模块，回答“遇到某类任务时应采用什么专门流程、知识和约束”。所以二者不是一回事，而是**同一控制思想在不同作用域上的实现**。

## 3. 把上下文窗口当记忆

能塞进模型上下文的文本，不等于记忆系统。记忆真正要处理的是持久化、检索、压缩、遗忘和跨轮次一致性。上下文窗口只是记忆被“临时读入”的通道之一。

## 4. 把 RAG 当长期记忆的全部

RAG 更像一种取回机制，不等于完整记忆系统。真正的长期记忆还包括写入时机、版本更新、冲突解决、个体隔离、时效性和删除策略。

note：“这块需要深入研究一下： RAG和长期记忆的特性及局限性， 使用策略”

补充回答：这块可以先抓一个实用区分。RAG 的强项是“读时取回”：面对文档库、知识库、政策库、代码库，它能把外部权威信息在需要时拉回来，优点是**新鲜、可溯源、易更新**；弱点是依赖检索命中率、切块质量和查询表达，而且它本身不负责“该不该写入”和“记住多久”。长期记忆的强项则是“跨会话保留”：用户偏好、长期项目背景、未完成事项、稳定事实，这些更适合写进 memory，而不是每次临时检索。它的难点在于写入污染、过期、冲突、隐私隔离和删除策略。比较稳妥的使用策略是：**权威知识优先用 RAG，用户特定且稳定的信息才进长期记忆；高风险事实把 memory 当线索，不当最终真相。** 再进一步说，长期记忆最好不要只做成一个向量库，还应区分 profile、episodic memory、task state、summary 等不同存储形态，并给每条记忆附上来源、时间、置信度和更新策略。

note+：“评估RAG系统质量的参数主要有哪些？ 检索命中率？ 切块质量？ 查询表达？”

补充回答：严格说，**检索命中率、切块质量、查询表达都重要，但它们不处在同一层。** 更稳妥的评估方式，是把 RAG 拆成“检索层、生成层、系统层”三层来看。

第一层是**检索层指标**，回答的是“该找回来的信息有没有找回来”。这层最常见的指标包括：`Recall@K` / 命中率（正确证据是否出现在前 K 个结果里）、`Precision@K`（取回结果里有多少是真的相关）、`MRR` / `nDCG`（相关结果排位是否足够靠前）。如果你的系统经常“明明知识库里有答案却没取出来”，优先看这一层。

第二层是**生成层指标**，回答的是“模型有没有基于证据正确作答”。这层常见的关注点包括：答案正确性、相关性、完整性、**faithfulness / groundedness**（回答是否真的被检索证据支持）、引用是否准确、该拒答时能不能拒答。很多 RAG 系统的问题不在“没检索到”，而在“检索到了但回答仍然幻觉”。

第三层是**系统层指标**，回答的是“这套 RAG 值不值得在真实环境里跑”。这里通常看延迟、成本、稳定性、时效性、鲁棒性、来源覆盖、对噪声和冲突证据的处理能力。一个实验室里答得准的 RAG，不一定在生产里可用。

而你提到的**切块质量**和**查询表达**，更准确地说属于这三层背后的关键影响因子。切块质量主要影响检索能否命中：块太碎会丢语义，块太大又会引入噪声；要看语义完整性、信息密度、重叠策略、是否把同一事实切断。查询表达则影响检索召回：用户原问题是否需要改写、展开、补实体、做多路查询，这些会直接影响 recall。换句话说，**切块和查询更像“可调旋钮”，命中率、正确率、faithfulness 更像“最终指标”。**

如果要抓一个最实用的最小评估集，我会优先看五个量：`Recall@K`、答案正确性、faithfulness、引用准确率、端到端延迟。因为这五项基本能覆盖“找没找到、答没答对、有没有胡说、能不能追溯、值不值得上线”这几个核心问题。然后再回头细查：问题究竟出在切块、查询改写、重排、还是生成阶段。

如果再把它变成一个实用的排查顺序，可以这样看：如果 `Recall@K` 很低，先查查询改写、embedding、切块和召回器；如果 `Recall@K` 不低但答案正确性差，优先查重排和生成提示；如果答案看起来像对的但 `faithfulness` 低，说明模型在“借题发挥”，要加强引用约束、上下文压缩和 grounded generation；如果质量还可以但延迟和成本失控，再回头调 `top-K`、重排链路和上下文长度。这样你就不会把所有问题都混成一句“RAG 效果不好”，而是能按层定位到底是哪一段出了问题。


note++：“这块有没有现成的评估技术手段,框架之类的”
有，而且截至 2026-03-16，这块已经不是“自己从零造轮子”的状态了。更准确地说，现成的不是一个万能框架，而是一套已经比较成熟的评估技术栈：离线数据集评估、LLM-as-
  judge、检索指标、trace 诊断、线上监控、人工复核，这几类通常组合使用。

  先给结论
  如果你只是想知道“现在有没有成熟工具能直接上手”，答案是：有，比较主流的就是 Ragas、DeepEval、TruLens、Phoenix、LangSmith，再加上 LlamaIndex 自带的评估模块，以及
  OpenAI 的 Graders / Trace grading 这类底层评估能力。
  但它们擅长的点不一样：

  - Ragas：更偏 RAG 专项离线评估，指标体系很全，官方文档直接列了 Context Precision、Context Recall、Faithfulness、Response Relevancy、Noise Sensitivity 等。适合
    先把“检索层/生成层”拆开测清楚。来源：Ragas Metrics (https://docs.ragas.io/en/stable/concepts/metrics/)、Context Precision
    (https://docs.ragas.io/en/stable/concepts/metrics/available_metrics/context_precision/)
  - DeepEval：更像 给 LLM/RAG 做 pytest。它的文档里直接提供 Contextual Precision、Faithfulness、Answer Relevancy、Hallucination 等单轮 RAG 指标，而且支持本地跑、
    远程跑、组件级和 E2E 级测试。适合接入 CI / 回归测试。来源：Confident AI / DeepEval 介绍 (https://www.confident-ai.com/docs)、Contextual Precision
    (https://www.confident-ai.com/docs/metrics/single-turn/contextual-precision-metric)、Faithfulness
    (https://www.confident-ai.com/docs/metrics/single-turn/faithfulness-metric)
  - TruLens：最适合做 RAG Triad 诊断。它把 RAG 评估压成三条主线：Context Relevance、Groundedness、Answer Relevance。如果你想快速判断“问题到底出在检索、 groundedn
    ess，还是回答相关性”，它很顺手。来源：TruLens RAG Triad (https://www.trulens.org/getting_started/core_concepts/rag_triad/)
  - Phoenix：更偏 生产级诊断 + tracing + evals。它官方文档明确支持 deterministic evaluators 和 LLM-as-a-judge，两种都能跑，而且能直接挂在 traces、datasets、实验
    结果上；对 RAG 它既能看 retrieval，也能看 response。适合“我不只想打分，我还想知道系统到底哪里坏了”。来源：Phoenix Evaluation
    (https://arize.com/docs/phoenix/evaluation/llm-evals)、Evaluate RAG (https://arize.com/docs/phoenix/cookbook/evaluation/evaluate-rag)
  - LangSmith：更偏 应用全生命周期评估平台。它不是只盯 RAG 指标，而是把离线评估、线上评估、datasets、runs、threads、LLM-as-judge、code evaluator、human evaluator
    都放进同一套工作流里。适合你已经在用 LangChain/LangGraph，或者你想把评估纳入长期开发流程。来源：Evaluate a RAG application
    (https://docs.langchain.com/langsmith/evaluate-rag-tutorial)、Evaluation concepts (https://docs.langchain.com/langsmith/evaluation-concepts)
  - LlamaIndex 自带评估模块：如果你本身就在这个生态里，它已经提供了 Faithfulness、Context Relevancy、Answer Relevancy、Correctness，也支持 LabelledRagDataset 和
    retrieval evaluation，还官方列了和 Ragas、DeepEval、RAGChecker 等的集成。适合“我不想再单独拼一套评估框架”。来源：LlamaIndex Evaluating
    (https://developers.llamaindex.ai/python/framework/module_guides/evaluating/)
  - OpenAI Graders + Trace grading：这不是“RAG 专用框架”，但如果你用 OpenAI 系生态，这是一套很强的底层评估积木。你可以自己定义 grader，用 reference answers、文本
    相似度、score model grader、甚至 Python grader 去评；trace grading 则很适合看 agent / RAG 的端到端轨迹。适合“我要自定义评估逻辑，而不是只吃预制指标”。来源：G
    raders (https://developers.openai.com/api/docs/guides/graders)、Trace grading (https://platform.openai.com/docs/guides/trace-grading)

  如果按“技术手段”而不是“产品名”来分，现成做法大概有 5 类

  - 经典检索评估：有标注时，用 Recall@K、Precision@K、MRR、nDCG、hit-rate 这类 IR 指标。这个最适合测 retriever，本质上不依赖 LLM judge。LlamaIndex 官方文档就明确
    把 retrieval eval 写成独立模块。
  - LLM-as-a-judge：现在最主流。拿另一个模型当裁判，测 faithfulness、groundedness、answer relevancy、context relevancy、hallucination。Ragas、DeepEval、Phoenix、
    TruLens 都在做这件事。
  - Reference-based / Reference-free 双模评估：有金标准答案时测 correctness / accuracy；没有时就测 groundedness / relevance。Ragas、LangSmith 都把这两类评估显式区
    分开了。
  - Trace-based 诊断：不只看最终答案，还看 retrieval、rerank、prompt、tool/output、span。Phoenix、LangSmith、OpenAI Trace grading、TruLens 都是这一类。
  - 线上评估 + 人工反馈：生产里只做离线评测不够，还要看真实流量。LangSmith 有 offline / online 的完整区分，Phoenix 也支持生产 traces 和线上 eval，Confident AI 还
    把 human annotation 单独做成工作流。

  如果你问“我该先用哪个”
  我会这样选：

  - 想最快起步，先把 RAG 指标体系跑起来：先用 Ragas
  - 想把评估接进测试/CI，像单元测试一样跑：先用 DeepEval
  - 想快速看 retrieval / groundedness / answer relevance 三角诊断：先用 TruLens

  真正可落地的推荐组合
  如果你不是在做论文，而是在做实际系统，我更建议这样搭，而不是只选一个框架：

  - 用 一小套人工整理的数据集 做离线基线评估
  - 用 IR 指标 测 retriever
  - 用 LLM-as-a-judge 测 faithfulness / answer relevancy
  - 用 trace 平台 看失败到底发生在哪一层
  - 对高风险场景保留 人工抽检或专家复核

  也就是说，框架只是执行器，真正重要的是你有没有把“检索错”“生成错”“引用错”“线上漂移”分开评估。

## 5. 把 MCP 当安全层、API 网关或“万能连接器”

MCP 解决的是交互标准化和能力协商，不等于权限治理，也不等于 API 生命周期管理。Google Cloud 在 S4 中明确指出，MCP 和 API 管理平台解决的是不同问题，它们是互补关系，不是替代关系。

note：“经典single responsibility”

补充回答：对，这里本质上就是 single responsibility。协议层的职责是“让不同能力以统一方式被发现、协商和调用”；安全层或 API 管理层的职责是“决定谁能调、能调多少、何时拦截、如何审计”。它们面对的是不同的变化源：协议会因互操作标准变化而变，安全会因组织策略和合规要求而变。把这两类责任揉在一起，后面一定会出现既难替换协议、又难独立治理安全的双重耦合。

## 6. 把多 Agent 当默认高级形态

多 Agent 的价值来自清晰分工，不来自数量本身。如果分工没有天然边界，只是把一个混乱问题拆成多个同样混乱的 Agent，那么你得到的通常不是更强系统，而是更难调的系统。

note：“这个tip不错， 质量 > 数量， 这块的设计也是需要深入研究一下的， 什么场景使用多agent， 我想这和场景以及吞吐量相关”

补充回答：对，而且判断多 Agent 该不该上，吞吐量只是其中一个变量，不是第一原则。更根本的问题是：任务能不能被切成**低耦合、可并行、可回收**的子问题。比较适合多 Agent 的场景通常有：大范围检索或调研、模块边界清晰的代码库探索、批处理评估、不同专业角色需要并行产出；不太适合的场景则是：上下文高度共享、决策持续互相影响、改一个地方会牵动全局的任务，比如大量重构或紧耦合编码。Anthropic 在 2025 年 6 月 13 日的研究文章里也明确提到，多 Agent 更适合 breadth-first 的开放式搜索，而很多 coding task 可并行的部分其实没有想象中那么多。简单说，**什么时候用多 Agent，不先看“任务有多大”，先看“边界有多清、并行值不值、汇总成本高不高”。**

## 7. 把“更多工具”当成“更强能力”

微软在 S9 里给出的反例很有代表性：工具过多、名字相近、响应过长、参数过深，都会让 Agent 更难选对工具。结构上，好的工具空间不是越大越好，而是越**清楚、互斥、可描述、可测试**越好。

## 8. 把护栏当成业务逻辑本身

护栏的职责是拦截、校验、降级、升级，而不是承担主要业务推理。如果把复杂业务规则全都塞进护栏，最后往往会得到一个又难维护又难解释的双重决策系统。

# 七、怎么吸收《通用Agent架构.md》的价值

旧文档最值得保留的，不是那张“八层图”本身，而是它试图表达的一件事：**Agent 永远是跨层系统。**

如果把旧文档里的层重新翻译成更有解释力的结构语言，大致可以这样看：

- 接入层、应用层：任务入口与会话承载面。
- 规划层、执行层：运行循环与任务推进控制。
- 记忆层：状态系统。
- 模型层：决策内核。
- 能力层：行动接口。
- 基础设施层：持久化、队列、缓存、日志、审计、策略。

这样吸收之后，旧文档的价值就不再只是“我知道 Agent 可以分八层”，而是变成：

1. 我知道每一层在消除什么失败模式。
2. 我知道哪些是核心结构，哪些是为了规模化而长出来的扩展结构。
3. 我知道真正该盯住的是控制权、状态流和边界治理，而不是层数本身。

这也是为什么学习 Agent 架构时，不能只背图。**图只是静态剖面，结构视角看的是控制关系和信息流。**

# 八、最后的判断标准

如果以后再看到一个“Agent 架构”，可以用下面四个问题快速判断它是不是讲到了结构本质：

1. 它有没有说清楚谁掌握下一步决策权？
2. 它有没有说清楚系统如何形成“计划-行动-观察-停止”的闭环？
3. 它有没有把状态系统和上下文窗口区分开？
4. 它有没有把工具、协议、治理、观测这些外圈结构分清，而不是混成一团？

如果这四个问题都答不清，那大概率只是一个功能堆栈图，不是 Agent 的结构视角。

从更高一点的位置看，Agent 的架构本质，其实是在做一件事：**把一部分原本属于人类操作者的“理解、选择、试错和调用外部能力”的过程，转移给模型来承担；与此同时，再用状态、协议、治理和观测把这部分转移出来的自主性重新收拢在可控边界内。**

这就是为什么 Agent 既不像传统工作流，也不像单纯聊天模型。它是一种介于“自动化系统”和“开放式决策系统”之间的结构形态。理解这一点，比记住多少层都重要。
