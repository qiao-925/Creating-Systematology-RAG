# Agent 的演化视角

如果把 Agent 直接理解成“LLM + tools + memory + loop”，很容易产生一个误解：好像系统一上来就该做成 Agent，仿佛这才是高级形态。更准确的看法其实相反。

Agent 不是起点，而是一层层失败模式逼出来的结果。每多长出一层能力，都不是因为名字更酷，而是因为上一层在真实任务里开始失效：单次调用不稳定，于是长出 workflow；模型缺事实和执行力，于是长出 tool use；固定流程不够用了，于是长出 loop；上下文装不下、跨会话断裂，于是长出 state 和 memory；工具能动真格以后，风险上升，于是 guardrails 成为主结构；单体塞不下职责边界时，multi-agent 才有理由出现；最终一旦面对真实用户和持续运行，生产化就不再是“部署一下”，而是整个系统目标的改变。

这份文档的主线只有一句话：**Agent 的演化，本质上是复杂度随着失败模式逐层上升。**

## Agent 为什么不是起点

先把一个常见误区掐掉：不是所有 LLM 系统都应该走向 agent。

Google Cloud 的官方架构文档写得很直接，像摘要、翻译、分类这类步骤预定义、结果边界清楚的任务，并不需要 agentic workflow。Anthropic 也明确建议先找到“最简单可行解”，复杂度只在需要时增加。OpenAI 对 agent 的定义同样非常克制：如果系统并不让模型去控制工作流执行，而只是单轮回答、简单聊天或分类，它就不算 agent。

note：“优先最简单可行解， 复杂度只在需要时增加， 这个让我联想到agentskill的渐进式披露， 其设计哲学不谋而合”

补充回答：这个联想是成立的。两者背后共享的是同一个复杂度治理原则：**不要预支复杂度，只暴露当前任务真正需要的那一层。** Agent Skills 的渐进式披露，解决的是“规则和知识不要一次性全塞进上下文”；这里强调的“先做最简单可行解”，解决的是“系统能力和架构不要一次性做满”。一个作用在认知输入侧，一个作用在运行结构侧，但本质上都在控制复杂度暴露面，避免系统在还没证明价值前就背上过重成本。

这背后的判断标准不是“是否用了 LLM”，而是两个更实在的问题：

1. 任务是不是天然多步，而且步骤之间存在不确定性？
2. 这个不确定性值不值得你用更高的延迟、成本、调试难度和风险去交换？

所以，Agent 不是“AI 系统的默认形态”，而是当传统确定性软件和单次模型调用都不够用时，才值得启用的一种更高复杂度结构。

## 先看整条演化主线

按复杂度治理而不是按流行词来看，比较清晰的一条线是这样的：

```text
单次调用
  -> workflow：因为一个 prompt 承载不了稳定多步任务
  -> tool use：因为模型缺少外部事实和执行能力
  -> agent loop：因为下一步不再能被预先写死，而要依赖环境反馈
  -> state / memory：因为任务跨回合、跨会话，且上下文窗口有限
  -> guardrails：因为系统一旦能行动，就会产生真实风险
  -> multi-agent：因为单体开始碰到专业、权限、上下文或并行边界
  -> 生产化：因为真实系统必须可观测、可评估、可暂停、可恢复
```

这里把 workflow 放在 tool use 前面，不是说所有团队都会严格按这个时间顺序演化。在现实里，很多系统会先接一个检索工具，再慢慢形成 workflow。之所以这里这样讲，是因为从设计判断上看，先问“任务结构是否确定”，通常比先问“要接几个工具”更重要。结构不清，工具越多，混乱只会越快到来。

## 第一步：单次调用是基线，不是落后

单次调用的价值，常常被低估。很多人一提 Agent，就默认单次调用代表“能力弱”。其实恰好相反：它是最干净的基线。

单次调用适合的问题通常具备几个特征：

- 输入边界清楚
- 输出目标明确
- 不要求跨回合连续行动
- 不依赖实时外部状态
- 错了也只是“答得不好”，而不是“做错了事”

这种形态的最大优点不是简单，而是**可评估、可控、可压缩**。你知道输入是什么，知道输出应该像什么，也更容易分辨问题到底出在提示词、模型选择，还是检索材料质量。

从演化视角看，单次调用阶段的任务不是“尽快升级到 agent”，而是先验证一件事：**问题到底是认知问题，还是系统问题。**  
如果一个任务通过更清晰的指令、结构化输出、少量检索增强就能稳定解决，那么它并没有真的提出 agent 需求。

Anthropic 在这里给了一个很重要的中间概念：**augmented LLM**。也就是说，系统可以已经接了检索、工具甚至记忆，但只要模型并没有被授权去控制整个工作流，它仍然更接近“增强过的模型调用”，而不是严格意义上的 agent。这个中间层非常关键，因为它让你避免掉“不是 agent 就只能是裸聊天”的二元误解。

note：“agent核心特征： 是否被授权一定的自主性”

补充回答：可以这么抓，但要补一句更精确的话：**自主性是必要条件，不是充分条件。** 更关键的不是“模型有多自由”，而是它是否被授予了**受限的工作流控制权**。也就是它能决定下一步、调用工具、根据反馈修正路径，并在失败时停止或把控制权交还给人。没有退出条件、责任边界和可验证反馈的“自主性”，本质上只是把不确定性放大，而不是真正的 agent 结构。

note+：“自主性是必要条件， 但是是受控制的自主性， 所以我并不认为这是全部的自主性， 完全的自主性意味着它可以做任何事情且不受人类控制， 受控制的自主性意味着它是在人类给定的范围内活动的有限自主性， 这块需要展开再收集一些资料， 目前对AI自主性的观点是如何界定的？”

补充回答：如果把这块再展开，当前更主流、也更可落地的看法，不是把“自主性”理解成有或没有，而是理解成**被授权范围内的自治程度**。工程上真正要问的，不是“它是不是完全自主”，而是四个更具体的问题：谁设定目标，谁决定下一步，谁批准高风险动作，谁在系统失控或失败时拥有最终停止权。

如果借用更偏标准 / 治理语境的表述，自主性通常都被理解为：**系统在被人类授予目标与自动化空间之后，能够在多大程度上无需持续人工介入地学习、决策或行动。** 这一定义里有两个关键限定。第一，自主性来自人类授权，不是凭空产生。第二，自主性是有程度差异的，不是非黑即白。

所以讨论 agent 的自主性，最好拆成几个维度来看：

- **目标自主性**：能不能自己提出子目标，还是只能执行人给定目标。
- **过程自主性**：能不能自己规划步骤、调整顺序、切换策略。
- **行动自主性**：能不能直接调用工具或触发外部副作用，是否需要审批。
- **恢复自主性**：失败后是自我修正、请求澄清，还是必须把控制权交还给人。

这样看，“完全自主”在生产里其实很少是默认追求，因为它会把责任边界、验证难度和安全风险一起推高。更现实的做法通常是分层监督：有的系统要求每一步都能被人工干预，有的是人主要做旁路监督，有的是人保留最终叫停权。自主性越高，guardrails、审计、评测和人工接管机制就越不能省。

## 第二步：Workflow 长出来，因为你需要稳定的多步结构

当一个任务开始失败，往往不是因为模型不聪明，而是因为你把太多不同性质的子任务塞进了同一个 prompt。

比如一次请求里同时要求“理解意图、提取事实、选择策略、生成执行结果、检查风险”。这些步骤并非不能由一个模型一次做完，但它们的成功标准不同，错误类型也不同。把它们揉在一起，输出就会变得难以预测，也难以定位问题。

这时 workflow 会长出来。它解决的不是“让模型更自主”，而是相反：**把原本埋在一次调用里的隐含步骤显式化**。

Anthropic 把这类模式讲得很清楚：prompt chaining、routing、parallelization，本质上都是工作流。开发者仍然掌握流程主导权，只是把“先做什么、后做什么、哪里做检查”从 prompt 内部搬到了系统结构上。这样做的收益很直接：

- 每一步有更清晰的输入输出契约
- 可以在中间插入程序化校验
- 可以对失败步骤单独评估和重试
- 可以让不同步骤使用不同模型、不同成本级别

所以 workflow 的出现，不是为了更炫的 orchestration，而是因为**你开始需要稳定的多步结构，而不是一次性押注模型把一切都想对。**

也是在这个阶段，很多团队第一次意识到一个朴素但重要的事实：所谓“智能感”，很多时候不是来自更大的 autonomy，而是来自更好的分工。

note：“经久不衰的分工思想， 从prompt到系统结构， 对可解释性和可靠性的工程追求”

补充回答：对，这里本质上就是“分工从 prompt 内部走向系统外部”。原来混在一个 prompt 里的理解、路由、执行、校验，逐步被拆成结构上可见的部件。这样做不是为了把架构画得更漂亮，而是为了让不同失败模式能被独立观察、独立替换、独立评测。换句话说，workflow 的出现并不是在追求更强的 autonomy，而是在追求更强的**可解释性、可维护性和可靠性**。

note+：“从工程下的角度评估， promot到系统结构， 其边界在哪里？ 在什么场景下需要由prompt到系统结构？ ”

补充回答：如果从工程角度划这条边界，最实用的判断不是“prompt 还能不能再写长一点”，而是**这个问题到底还属于一次性认知约束，还是已经变成了需要显式管理的系统行为。** 前者优先留在 prompt，后者就应该上升为系统结构。

比较稳妥的分界方法可以这样抓：

- 还适合留在 prompt 里的，通常是：单轮即可完成；输入输出契约相对简单；失败主要表现为“理解偏了”或“措辞不好”；不需要中间状态、外部审批、重试或分支控制。
- 需要从 prompt 升到系统结构的，通常是：任务已经变成多步；不同步骤的成功标准不同；需要工具调用、状态持久化、分支路由、重试、预算控制、审批门或程序化校验；你开始需要知道“它为什么失败、失败在第几步、能不能只修这一段”。

可以把它理解成一句更短的话：**只影响“怎么想、怎么说”的规则，优先留在 prompt；一旦开始影响“怎么流转、怎么执行、怎么治理”，就应该进入系统结构。**

典型场景包括：

- **从 prompt 到 workflow**：当一个请求里混着理解、检索、生成、校验，且这些步骤需要拆开评估时。
- **从 prompt 到 tool contract**：当模型必须稳定调用外部能力，参数、返回值和副作用都需要明确时。
- **从 prompt 到 state management**：当任务跨回合、跨会话，或者上下文窗口已经不足以自然承载连续性时。
- **从 prompt 到 guardrails / policy**：当错误不再只是“说错”，而会变成“做错”“写错”“泄露错”时。

所以真正的边界，不是 prompt 能不能继续承载，而是**继续把问题塞进 prompt，会不会让可解释性、可测试性和可治理性开始明显下滑。** 一旦答案是会，就该把这部分责任从语言层移到系统层。

note++：”简要概括为几点，最核心的就是复用和复杂度评估，一次性约束不值得使用复杂的系统结构，相对简单的问题也不用上复杂的方案，但是对于复杂，稳定要求的任务，可以逐步从语言层到系统层过渡。“

补充回答：可以，把这段再压一层，最核心其实就是三句话：

- **一次性、低风险、低复用的约束，优先留在 prompt。** 这类问题主要是表达和理解问题，不值得为了它引入额外系统复杂度。
- **可复用、可拆步、需要稳定性的约束，优先升到 workflow 或工具契约。** 因为你已经不只是想“这次答对”，而是想“以后也能稳定跑对”。
- **涉及状态、权限、审批、回退、审计和安全边界的约束，必须升到系统层。** 因为这些已经不是语言提示能可靠承担的职责，而是工程治理职责。

所以更短地说，判断边界时就看三件事：**是否复用、是否多步、是否高风险。** 一次性问题尽量留在语言层；重复性问题逐步外移到结构层；高风险问题直接纳入系统治理层。

## 第三步：Tool Use 长出来，因为模型既缺事实也缺手脚

只靠模型参数，系统很快会碰到两类边界。

第一类边界是**事实边界**。模型知识有截止日期，实时信息会过时，专有数据本来就不在预训练里。  
第二类边界是**行动边界**。模型能生成文本，但默认并不能查数据库、发请求、改工单、跑代码、下订单。

这就是 tool use 长出来的原因。OpenAI 的工具文档把这个边界说得很直白：工具让模型获得 web search、function calling、file search、MCP 等外部能力。Toolformer 这类代表性论文则进一步指出，语言模型在算术和事实查找这类任务上，本来就存在结构性短板，借助外部工具不是锦上添花，而是补齐能力边界。

因此，tool use 的真正含义不是“让模型会调 API”，而是把系统从：

- 一个只能基于内部表征回答的生成器

变成：

- 一个能够判断何时应访问外部世界、该调用什么能力、如何消费返回结果的决策器

这一步一旦发生，系统性质就开始改变了。因为从现在开始，错误不再只是“答非所问”，还可能变成“查错地方”“调用错工具”“把不该传的数据传了出去”。

也正因为如此，工具设计本身往往比 prompt 更重要。Anthropic 的经验非常现实：工具接口越清晰、文档越明确、参数越少歧义，agent 表现越稳定。换句话说，**很多所谓 agent 问题，本质上其实是工具工程问题。**

note：“工具工程！！！ tool： 模型的手和脚； 从当前的演化路径来看， agent非常的拟人化， 模拟人的思考， 模拟人的行动”

补充回答：这个比喻很有启发，但工程上最好保留一点克制。tool 确实像模型的“手和脚”，让它从“只会说”变成“能查、能做、能影响外部世界”；不过它不是自然长出来的器官，而是**显式定义的操作面**，带着 schema、参数约束、权限边界和副作用。更准确地说，agent 在结构上越来越像“感知 - 判断 - 行动 - 反馈”的闭环系统，而不是真的在复制人类。拟人化有助于建立直觉，但设计时仍要回到接口、状态和治理这些工程事实。

## 第四步：Agent Loop 长出来，因为下一步开始依赖环境反馈

Workflow 解决的是“多步，但步骤基本可预定义”。真正让系统走向 agent 的分水岭，通常不是用了几个工具，而是**下一步做什么，已经不能提前完全写死**。

一个典型例子是搜索、调试、调查、故障排查这类任务。系统需要先行动一下，看到结果，再决定下一步：

- 搜索结果不对，就换关键词
- API 报错，就读错误信息再改参数
- 测试失败，就根据失败日志修正方案
- 页面信息不足，就继续浏览下一层

这时，固定 workflow 就开始吃力，因为核心难点不在步骤数量，而在**步骤之间存在环境反馈**。

ReAct 给出的洞察正好解释了这里为什么会长出 loop：模型需要交替进行 reasoning 和 acting，并把观察结果重新带回推理。OpenAI 在 agent 指南里也明确说，while loop 是 agent 的中心概念。Anthropic 的表述更进一步：很多 agent 的本质，不过是“LLM 在循环中基于环境反馈调用工具”。

这一步的关键变化是，系统从“执行预定义流程”变成“在执行过程中持续决定流程”。

这很强大，但也很危险。因为 loop 一旦出现，新的失败模式也会一起出现：

- 反复调用无效工具
- 在错误方向上越走越远
- 成本和延迟失控
- 局部错误在循环里被放大

所以 loop 不该因为“agent 就该有 loop”而引入，而应该在两个条件同时满足时才引入：

1. 下一步确实依赖环境反馈，而不是你偷懒没把流程想清楚。
2. 成功与失败存在外部可验证信号，系统可以知道该停、该改、还是该交还给人。

这也是为什么编码、故障排查、研究式搜索特别适合 loop：它们通常有测试结果、执行结果、搜索结果这类可反馈信号。

note：“这让我联想到系统之美中提到的正反馈和负反馈， btw， loop模式和人在做某件事情尝试不同的方法超级类似， 反馈在其中扮演核心角色”

补充回答：这个类比非常准。loop 本质上就是一个反馈系统。正反馈负责放大有希望的方向，例如搜索命中了有效线索后继续深挖；负反馈负责抑制发散，例如测试失败、预算耗尽、证据矛盾、guardrails 拦截后要求回退。人类做研究、调试、排障时其实也是这样：先试一个方向，再根据结果修正路径。真正好的 agent loop 不是“能一直转”，而是**能在反馈作用下逐步收敛**。从工程角度看，stop condition、预算上限、审批门和校验器，本质上都是给 loop 补上必要的负反馈。

## 第五步：State / Memory 长出来，因为任务不再只活在一个 Prompt 里

只要系统进入多步循环，状态问题就躲不掉。

OpenAI 的状态文档提醒了一个基础事实：单次文本生成请求天然是无状态的。你之所以觉得聊天“有上下文”，往往只是系统把历史内容又拼回去了。可一旦进入真实 agent 场景，状态不只是聊天历史，还包括：

- 已经做过哪些步骤
- 调过哪些工具
- 工具返回了什么
- 当前计划进行到哪
- 哪些约束已经被触发

也就是说，state 不是“顺便存一下聊天记录”，而是 agent 执行轨迹的一部分。

这里最好把两种东西分开看：

### 1. 短期状态

短期状态服务于当前任务执行，例如当前会话中的消息、工具调用、工具输出、阶段性计划、失败次数等。OpenAI 的 Conversations API 明确把这些都视为会话状态的一部分。

短期状态长出来，是因为 loop 需要连续性。没有它，系统每一轮都像失忆重来。

### 2. 长期记忆

长期记忆服务于跨会话连续性，例如用户偏好、长期项目背景、历史决策、可复用知识。Google Cloud 的架构文档把它看作持久化知识层，并强调在生产场景里往往要用外部存储管理，而不是塞在单实例内存里。

长期记忆长出来，不是因为“高级 agent 都有 memory”，而是因为你真的存在跨会话、跨设备、跨任务的连续性需求。

很多系统在这里最容易过度设计。一个常见错误是：明明只是当前任务上下文太乱，却急着上“长期记忆”。这通常是在拿持久化复杂度掩盖任务建模问题。  
更稳妥的顺序应该是：

- 先把短期状态理顺
- 再处理上下文压缩和摘要
- 只有当跨会话连续性真正创造价值时，才引入长期记忆

否则，memory 不会让系统更聪明，只会让它更难治理、更难清理、也更容易带入陈旧或错误信息。

note：“只有当跨会话连续性真正创造价值时，才引入长期记忆， 这句给了我另一个启发， 关于状态升级， 对话也好， 其他事物也好， 本质上需要做分层处理， 例如基础的信息层， 类似日志， 而在日志中累积起来的经验和可复用的知识， 则可以抽象为更高一级的规则，规范， 例如agentskill这类的可复用的技能”

补充回答：这个启发很有价值，而且可以进一步抽象成一条“状态升级链”：**原始事件 / 日志 -> 会话状态 -> 摘要与中间结论 -> 稳定偏好与规则 -> 可复用 skill。** 日志层提供证据，状态层保证连续性，摘要层负责压缩，规则层负责跨任务迁移，而 skill 则是经过验证后沉淀下来的高阶能力模块。这里最重要的不是“能不能升级”，而是“升级门槛是什么”。不是所有高频片段都值得升格为 skill，只有那些跨任务重复出现、相对稳定、可验证、且收益明显的模式，才值得从经验变成规则。

## 第六步：Guardrails 长出来，因为“会行动”就会带来风险

一个只能回答问题的系统，最常见的失败是“说错”。  
一个能调用工具、能写入外部系统、能持续循环的系统，失败会升级成“做错”。

这就是 guardrails 必然长出来的原因。它不是锦上添花，而是行动能力的配套成本。

OpenAI 的安全文档把风险说得很具体：prompt injection、私有数据泄露、MCP 工具调用风险、非预期行为。OpenAI 的 agent 指南还补了一层很重要的工程判断：高风险、不可逆、涉财务或权限敏感的动作，应触发人工介入；同时要设置失败阈值，超过重试上限就把控制权交还。

因此 guardrails 的核心价值，并不是“让模型更安全一点”，而是让系统具备以下能力：

- 知道哪些输入不可信
- 知道哪些输出不能直接传下游
- 知道哪些工具调用必须审批
- 知道什么时候应该停止循环
- 知道什么时候必须交还给人

从演化视角看，guardrails 不是最后补上的围栏，而是 agent 获得真实行动力之后，与工具、状态、权限一起同步长出来的主结构。

如果一个系统已经能写库、发单、删数据、调资金，但 guardrails 还是“后面再说”，那不是在做 agent，而是在把不确定性直接暴露给生产环境。

note++：”安全性的考量在AI语境下被放大， 目前有哪些常见的方案？“

补充回答：如果按当前比较主流的工程实践来归纳，AI / agent 的安全方案本质上不是单点能力，而是**分层防线**。OpenAI 当前的 agent safety 文档、agent guide，以及 Microsoft 的多 agent 架构指导，基本都在强调同一个思路：不要指望某一个 guardrail 一次性解决问题，而要把安全拆到输入、执行、权限、输出和运行治理几个层面。

常见方案大致可以分成下面几类：

- **输入防线**：检测 jailbreak、prompt injection、PII、恶意内容和不可信外部文本。核心目标是不要让未验证数据直接进入高权限上下文。
- **结构化数据流**：尽量用 schema、枚举、固定字段、结构化输出在节点之间传值，而不是把自由文本直接喂给下游工具或下游 agent。这是当前抑制 injection 传播最有效的常见方法之一。
- **最小权限与工具审批**：工具按最小权限开放，高风险读写默认审批，人机确认保留在关键动作前。尤其是写操作、付款、删除、发信、提交外部系统这类动作，不应该完全裸放给 agent。
- **执行期 guardrails**：在工具调用前后、模型中间结果和最终输出上做校验，例如关键词 / 规则过滤、相关性校验、安全分类、敏感信息拦截、越权检查、格式校验。
- **Human-in-the-loop**：对高风险、不可逆、超预算、连续失败或低置信度情形，显式交还给人。这不是“系统失败后的补救”，而是生产级 agent 的标准设计。
- **预算与熔断**：限制最大步数、最大重试次数、最大 token / 成本、最大工具调用次数；一旦超过阈值就停止、回退或升级处理。否则 loop 会把安全问题和成本问题一起放大。
- **隔离与沙箱**：对工具执行环境、agent 间通信、知识库访问和外部网络访问做隔离，避免一个环节被污染后横向扩散。多 agent 场景里这点尤其重要。
- **可观测与评测**：保留 trace、审计日志、失败记录，并持续做 evals / trace grading。安全不是一次配置，而是靠持续观测和复盘迭代出来的。

如果再把这些方案压成一个更易记的框架，我会写成：**先拦输入，再限通道，再控权限，再审执行，最后保留人工接管和审计闭环。** 这也是为什么 AI 语境下的安全会被放大，因为模型既会理解输入，又会调用工具，还可能持续循环；一旦其中任一层失守，风险就会沿着整条链条传播，而不是停留在“一次回答说错了”。

## 第七步：Multi-Agent 长出来，因为单体开始碰到真正的边界

多 Agent 很容易被过度想象。很多人一看到 planner、reviewer、executor 这类角色，就自然想把它们拆成多个 agent。Microsoft 的官方建议恰好反过来：角色分工并不自动意味着要做 multi-agent，很多时候一个 single-agent 配合 persona、条件提示、权限控制就够了。

所以 multi-agent 该在什么情况下长出来？通常是因为单体确实碰到了以下边界之一：

- **专业边界**：不同子任务需要明显不同的上下文和策略，混在一个 agent 里会相互污染。
- **权限边界**：不同动作需要不同安全域，不能让一个 agent 拿到所有权限。
- **上下文边界**：一个 agent 同时背负太多工具、知识源和状态后，决策质量开始下降。
- **并行边界**：任务天然适合并行分治，拆开能显著降低整体时延或提升质量。

这时，多 Agent 才是在解决真实结构问题，而不是扮演“更高级架构”。

但它带来的成本必须正视。Microsoft 的两篇官方文档都反复强调，多 Agent 会增加：

- 协调与路由开销
- 上下文传递复杂度
- 延迟和 token 成本
- 新的失败模式
- 更难的安全与可观测性问题

所以更成熟的判断不是“我们是否能做 multi-agent”，而是：**单 Agent 是否已经被证明不够。**

这也是为什么 OpenAI 和 Microsoft 都倾向于同一个默认顺序：先把 single-agent 的能力压到尽头，再决定是否升级成 multi-agent。

note：“关于并行的使用体验， codex由于设计的质量优先考量， 普遍任务的响应时间比较高， 所以在日常使用中， 发现当前任务有可能设计为并行执行， 都会在提示词中要求以加速执行，减少等待； 再就是前几天claude code退出的多agent review代码的功能， 用起来也是效率极高以及质量非常不错”

补充回答：你的体感和当前公开实践基本是对得上的。并行真正有价值的前提，不是“agent 数量多”，而是任务满足**低耦合、可并行、结果可压缩回传**。以当前 Codex 的使用体验看，把代码库摸查、独立测试、信息搜集这类 side task 并出去，通常确实能减少主线程等待；但如果任务高度共享上下文、需要频繁来回讨论，盲目并行反而会增加集成成本。Claude Code 的 code review 之所以体验好，不只是因为“多 Agent”，更因为官方公开文档里明确写到它采用一组 `specialized agents` 并行找问题，再用 `verification step` 过滤误报。也就是说，它把“并行发现”和“验证降噪”拆成了两个阶段，这才是质量和效率同时上来的关键。

note+：”上下文依赖程度是衡量并行效率的核心参数！验证降噪这块展开说说claude code的实现？ 有什么工程启发？“

补充回答：这块如果只根据 Claude Code 公开资料来还原，可以确认的机制有三层。第一层是**并行发现**：官方文档明确写到多个 agent 会并行分析 diff 及其周边代码，而且关注不同类型的问题。第二层是**验证降噪**：不是把候选问题直接发出来，而是再走一个 verification step，用更贴近实际代码行为的检查去过滤误报。第三层是**结果治理**：最终结果会去重、按严重级别排序，并挂到具体行内，而不是把一堆模糊警报平铺出来。

如果再结合 Anthropic 对外披露的实现细节，可以把它理解得更具体一些：它不是让多个 reviewer 泛泛地“都来看看”，而是让不同 reviewer 带着不同视角工作，例如 bug、合规、历史上下文、过往评论、注释一致性等，然后只把高置信度问题送到最终输出层。这里我做一个明确推断：**它的关键不是多 agent 本身，而是“候选发现”和“证据验证”被拆成了两个不同阶段。**

这个设计给工程上的启发很直接：

- **并行适合做候选生成，不适合直接做最终裁决。** 多个 agent 可以并行扩大覆盖面，但最终输出最好经过统一验证和排序。
- **验证必须尽量靠近真实行为。** 对代码 review 来说，最有价值的不是“听起来像问题”，而是能不能在代码逻辑、调用路径、边界条件或历史约束里找到更硬的证据。
- **上下文依赖程度确实是并行效率的核心参数。** review 场景之所以适合并行，是因为每个 reviewer 共享同一个代码事实面，但可以从不同缺陷模型出发独立分析，最后回收成较小的 finding 集；这比并行写代码更容易收敛。
- **降噪不是最后润色，而是架构层职责。** 如果没有 verification step，多 reviewer 往往只会放大重复告警和伪阳性，结果是人类 review 更累，而不是更轻松。

所以这套实现最值得学的，不是“开更多 agent”，而是把多 agent 的工作链拆成：**并行发现 -> 证据验证 -> 去重排序 -> 人类消费。** 这其实是一个非常典型的生产级质量控制结构。

## 第八步：生产化长出来，因为“能跑通”不等于“可运营”

很多 agent 讨论在 Demo 阶段就结束了，但真正麻烦的部分其实是上线之后才开始。

在 Demo 里，你关心的是“这次是否成功”。  
在生产里，你关心的是：

- 它是否稳定成功
- 失败时是否可诊断
- 成本是否可预测
- 延迟是否可接受
- 状态是否可恢复
- 风险是否可审计
- 质量是否可持续改进

这时，系统目标已经变了。它不再只是“让模型完成任务”，而是把一个带随机性的推理-行动系统变成**可运营的服务**。

Google Cloud 的架构文档在这里非常有启发性：生产级 agent 往往要求外置化状态管理、健壮的后端、可扩展的会话管理。OpenAI 的 evals 文档则指出，agent 质量必须通过可复现实验来衡量，而不是靠几次漂亮演示。也就是说，生产化之所以会长出来，不是因为“规模大了要部署”，而是因为你终于开始承认：**agent 是一个需要观测、评测、治理和恢复机制的系统，而不是一个会说话的功能。**

所以生产化阶段真正长出来的，通常不是一个新 prompt，而是这些能力：

- trace 与日志，用来理解 agent 到底做了什么
- evals，用来衡量质量是否稳定
- 预算与限额，用来防止 loop 把成本拉爆
- 超时、重试与回退，用来应对外部工具不稳定
- 外部状态存储，用来支持长任务和跨实例恢复
- 审批与审计，用来覆盖高风险动作

这也是为什么“生产 agent”从来都不是某个框架开关，而是整个工程重心的转移。

note：“agent生产级的重工程设计也是人类在未来为数不多的生存空间， 可观测（日志）， 质量评估（跑分）”

补充回答：这句话如果收敛一下，我会表述成：**人类的高价值会越来越集中在“定义系统边界和评估系统行为”上。** 模型可以承担越来越多的生成、检索、调用和执行，但什么叫成功、哪些风险不能接受、哪个 trace 值得升级排查、eval 应该测什么、阈值该设在哪，这些依然是强工程判断和价值判断。可观测性、评测、预算、回退、审计这些能力，实际上是在把人类经验从一次次手工救火里抽出来，沉淀成长期可运营的反馈系统。

note+：“手动，重复的机械工作时代慢慢过去，判断和价值评估的高密度工作时代即将到来！”

补充回答：这句话的大方向是对的，但如果写进文档，我会把它说得更工程化一些：**重复、规则明确、可局部验证的执行工作会持续被自动化压缩，而人类价值会越来越集中到目标设定、边界划定、结果评估和异常处置上。** 也就是说，不是“人不再工作”，而是工作重心从“亲手做每一步”转向“判断哪些步骤该交给系统，以及系统做得是否可接受”。

更具体一点，未来更稀缺的高密度工作通常包括：

- 定义成功标准，而不是只产出结果
- 判断风险边界，而不是只追求自动化率
- 设计评测与验收机制，而不是只看 demo 效果
- 在冲突证据、不完整信息和多目标权衡中做取舍
- 把零散经验沉淀成可复用的 policy、skill 和工作流

所以“判断和价值评估”之所以更重要，不是因为它更抽象，而是因为 agent 越强，系统越需要有人决定三件事：**什么值得做，做到什么程度算好，出了问题由谁接住。** 这三件事目前都还高度依赖人的业务理解、责任意识和长期视角。

## 哪些复杂度不要过早引入

如果把前面的演化线倒过来看，就会得到一组很实用的克制原则。

### 1. 不要用 Agent 处理本来就是确定性的任务

如果摘要、翻译、提取、分类已经足够，agent 往往只会引入更高成本、更长时延和更差可调试性。

note：“想到一个信息收集case， agent擅长处理一定的不确定性， 刚好可以作为自主决策的智能工具， 去做某个领域的信息收集， 处理， 洞察； 给予一定的自主性， 如信息源的选择， 当然需要限定在相对可靠，可信， 官方的来源， 但最终的判断交给ai， 这样可以极大扩展当下热议的信息茧房问题”

补充回答：这是 agent 很适配的一类场景，尤其是**目标明确，但搜索路径和证据组合方式不确定**的信息收集 / 研究型任务。它的优势在于：可以自主决定先查哪里、什么时候换关键词、哪些来源值得继续深挖、何时做交叉验证，再把结果压缩成结构化洞察。不过这里要区分“自主搜集”与“自主裁决”。来源选择可以在可信来源池内给 agent 一定自由，但最好同时给它来源分级、时效窗口、引用要求和冲突处理规则；最终结论如果涉及高风险判断，仍应保留人工复核。至于信息茧房，agent 的确有机会缓解，因为它可以被要求主动覆盖不同来源与相反证据；但如果来源池、排序目标或提示本身带偏，它也会把偏见自动化放大。所以设计上最好要求它显式呈现来源分布、冲突证据和不确定性，而不是只给单一路径的结论。

note+：”这是细化的设计思路，非常不错，偏见的自动化放大非常值得警惕，做可信度评估会是非常重要的上下文，这块可以系统的展开一下，当前上下文需要首要考虑的三个问题是什么？“

补充回答：如果把这类“研究型信息收集 agent”系统化设计，当前上下文里最该先想清楚的不是模型提示词，而是下面三个问题。顺序最好也就是这个顺序：

1. **它被允许相信谁？**  
   也就是来源边界怎么划。可信来源池是否预先限定，来源要不要分级，官方 / 一手 / 二手 / 社区材料权重怎么不同，时效窗口怎么设，是否要求至少两类相互独立来源交叉验证。很多所谓“偏见问题”，其实首先是来源治理问题。

2. **它如何处理证据冲突？**  
   现实信息常常不是一致的。系统不能只做“搜到最多的那个说法就是结论”，而要有一套冲突处理规则，例如记录来源、比较时效、区分事实与评论、显式展示分歧、给出不确定性说明。否则 agent 只是在自动化放大最先命中的叙事。

3. **它最终被授权输出到什么程度？**  
   是只做资料整理、证据汇总和候选洞察，还是可以直接给判断、建议，甚至触发后续动作？这一层决定了你需要多强的人工复核、置信度阈值和 guardrails。越接近“裁决”或“行动”，就越不能只靠来源覆盖，而必须引入更严格的审核机制。

如果把这三点压成一句工程化表述，就是：**先治理来源，再治理证据，再治理裁决权。** 这也是避免“偏见被自动化放大”的最核心顺序。因为只要前两层没管住，最后一层看起来再谨慎，结论也可能只是被包装过的偏见。

### 2. 不要因为“有多个角色名”就直接上 Multi-Agent

planner、reviewer、executor 这些概念经常只是 prompt 里的角色分工，不一定需要拆成多个自治体。

note：“这里要深入研究一下， 多agent的分界点在哪？ 目前是否有可参考的技术指南”

补充回答：多 Agent 的分界点，不是“任务听起来很复杂”，而是**single-agent 优化之后，仍然存在结构性瓶颈**。比较实用的判断标准有四条：一，单 Agent 因工具或上下文过载而持续掉质；二，任务天然可并行，且子结果容易汇总；三，权限 / 安全域必须分离；四，不同子任务需要明显不同的 prompt、工具和评价标准。如果这些边界都还不明显，通常先留在 single-agent 更好。至于技术指南，目前官方可参考资料已经不少：OpenAI 的《A practical guide to building agents》、Microsoft 的《Choosing Between Building a Single-Agent System or Multi-Agent System》与《AI Agent Orchestration Patterns》、Anthropic 的《How we built our multi-agent research system》，以及 Claude Code 的 `subagents`、`agent teams`、`code review` 文档。它们的共同结论非常一致：**先把单 Agent 做到位，再用多 Agent 解决真实边界，而不是为了架构好看而拆分。**

note+：“这块系统分析一下上文提到的技术指南中的设计哲学”

补充回答：如果把上面提到的几份官方技术指南放在一起看，它们虽然来自不同公司，但设计哲学其实高度一致。真正的共识不是“agent 很强”，而是下面这几条：

- **复杂度后置，不要预支。** OpenAI 强调 incremental approach，Anthropic 强调从 augmented LLM 和 workflow 逐步长到 agent，Microsoft 强调先做 single-agent test。共同点都是：先用最小可行结构验证价值，再为真实失败模式增加复杂度。
- **结构服务于失败模式，不服务于概念包装。** 什么时候上 workflow、multi-agent、guardrails，不是看架构图好不好看，而是看当前失败到底来自步骤混杂、工具不稳、状态混乱、权限冲突，还是上下文和并行边界。
- **单体优先，拆分要有硬理由。** 几家资料都反复提醒，多 Agent 只有在专业边界、权限边界、上下文过载或并行收益已经被证明时才值得引入。
- **并行不是默认答案，信息压缩才是关键。** Anthropic 和 Claude Code 都强调并行要建立在子任务低耦合、结果可回收、上下文能隔离的前提上；否则通信和集成成本会吃掉收益。
- **工具、权限、状态、评测是主结构，不是配件。** OpenAI 更强调 guardrails、human intervention 和 evals；Microsoft 更强调架构模式、协调成本和治理；Claude Code 文档更强调上下文隔离、权限收敛、任务分配和验证降噪。但本质上都把这些看成系统主体，而不是 prompt 外围装饰。

如果再看它们各自的侧重点，也很有意思：

- **Anthropic** 更像是在教你如何控制复杂度上升节奏，核心词是 augmented LLM、workflow、context、tool design。
- **OpenAI** 更像是在教你如何把 agent 做成可运行、可接管、可治理的产品系统，核心词是 loop、guardrails、human intervention、evals。
- **Microsoft** 更像是在做企业架构判断，核心词是 security boundary、coordination overhead、orchestration pattern、organizational fit。
- **Claude Code 文档** 更像是在把这些理念翻译成可直接操作的产品机制，核心词是 context isolation、permission boundary、parallel work、verification。

所以如果要把这些指南的共同设计哲学压成一句话，我会写成：**agent 不是“更聪明的 prompt”，而是“围绕不确定性、权限和失败模式逐层加出来的系统结构”。** 这也是为什么这些资料最后都会收敛到同一个判断：先证明单体不够，再升级协作；先让系统可控，再让系统更自主。

### 3. 不要在工具还不稳定时追求自主循环

工具接口不清、返回不稳、权限不清楚时，loop 只会把错误放大。很多“agent 不靠谱”其实是“工具系统没打磨好”。

### 4. 不要在短期状态还混乱时急着上长期记忆

如果当前任务轨迹都管理不好，长期记忆只会让系统把混乱永久保存下来。

### 5. 不要把 Guardrails 当作上线前最后一层补丁

一旦系统涉及写操作、财务动作、敏感数据或跨系统权限，guardrails 就应该和工具一起设计，而不是后补。

### 6. 不要把框架复杂度误当作系统成熟度

图编排、节点 DSL、复杂编排平台当然有用，但只有在问题真的需要时才值得承担。很多团队真正需要的，是更清晰的任务边界、更好的工具接口和更严格的 evals。

## 结语：把 Agent 当作复杂度预算，而不是默认答案

从演化视角看，Agent 从来不是一个孤立名词，而是一条不断上升的复杂度阶梯。

你不该问“我们要不要做 Agent”。更好的问题是：

- 现在的失败模式到底是什么？
- 这个失败模式属于结构问题、能力问题、状态问题，还是风险问题？
- 为了解这个问题，最小必要复杂度是多少？

如果你用这组问题去看系统演化，就会发现一件事：真正好的 Agent 设计，通常不是最自主、最花哨、最像人类团队的那个，而是**在刚好足够的复杂度上，稳定地解决了原本解决不了的问题**。

这才是“演化视角”最重要的洞察：  
Agent 不是目的，**它只是你为处理更高层级不确定性而支付的复杂度预算。**
