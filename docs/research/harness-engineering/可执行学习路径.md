# Agent 开发可执行学习路径

## 1. 先回答一个关键问题

### 跟着 Codex 和 Claude Code 去了解 Agent 开发，合适吗？

合适，但只适合作为入门加速器，不适合作为唯一学习路径。

如果你的目标是先快速建立对 Agent 开发的直觉、工作流和工程感，跟着 `Codex` 和 `Claude Code` 学很有效。因为它们能把很多抽象概念变成可观察行为，例如：

- 如何接收目标并拆分任务
- 如何读取上下文并决定下一步
- 如何调用工具
- 如何在失败后恢复
- 如何持续汇报进度并交付结果

但如果你的目标是系统掌握 Agent 开发，只跟着这类成熟产品不够，主要原因有三个：

1. 你能看到现象，但看不到很多关键内部机制，容易停留在“会用”而不是“会做”。
2. 你容易过早沉迷效果演示，而忽略状态管理、失败恢复、评估体系、安全边界这些真正决定工程质量的部分。
3. 你可能学成“会调 Prompt、会接工具”，但不会设计一个可维护、可验证、可扩展的 Agent 系统。

结论可以压缩成一句话：

- 想快速入门：合适
- 想系统掌握：不够
- 最佳用法：边观察成熟 Agent，边自己复刻最小版本

## 2. 用 Codex 和 Claude Code 应该重点学什么

把它们当作“实战参照物”，重点观察下面四类东西：

### 2.1 Agent 的基本交互范式

- 如何接收任务
- 如何拆解任务
- 如何决定使用工具还是直接回答
- 如何在中间步骤汇报进展
- 如何把执行结果整理成最终交付

### 2.2 Coding Agent 的工程闭环

- 如何读代码
- 如何定位修改点
- 如何执行命令和测试
- 如何处理失败
- 如何继续迭代直到结果稳定

### 2.3 Prompt 与 Tool Schema 的设计直觉

- 哪些信息应该放在 system prompt
- 哪些约束适合放进工具参数
- 哪些规则应该由运行时强制保证
- 为什么有些事情适合结构化输出，有些适合自然语言

### 2.4 人机协作边界

- 什么适合让 Agent 自动完成
- 什么必须保留给人做决策
- 什么场景需要确认
- 什么场景应该直接执行

## 3. 不能只跟着成品 Agent 学的部分

下面这些内容不能只靠观察 `Codex` 或 `Claude Code` 获得，必须自己补：

### 3.1 Architecture

- 单 Agent 还是多 Agent
- Planner-Executor 还是 Supervisor-Worker
- 各模块边界怎么划分

### 3.2 Memory

- 短期上下文怎么管理
- 长期记忆怎么存
- 历史内容怎么裁剪与摘要
- checkpoint 如何恢复

### 3.3 Evaluation

- 怎么定义任务成功
- 怎么衡量轨迹质量
- 怎么判断工具调用是否正确
- 怎么做自动评估

### 3.4 Reliability

- 幻觉怎么约束
- 死循环怎么检测
- 工具调用失败怎么恢复
- 上下文漂移怎么处理

### 3.5 Safety

- 命令执行权限怎么隔离
- 文件访问范围怎么限制
- 联网策略怎么定义
- 敏感数据怎么处理

## 4. 一条更靠谱的学习顺序

### 第一步：先观察成熟 Agent 的行为

先不要急着做复杂系统，先观察一段时间。重点不是看它“有多强”，而是记录它每一步在做什么。

你至少要能观察出这些问题的答案：

- 它什么时候先读环境
- 它什么时候先出计划
- 它什么时候直接执行
- 它什么时候调用工具
- 它什么时候停下来向用户确认
- 它失败后如何恢复

### 第二步：自己复刻一个最小 Agent

不要一开始就做多 Agent。先做一个最小可运行版本，只保留：

- 一个模型
- 一个 Planner
- 一个 Orchestrator
- 两三个工具
- 一个简单 Memory
- 一个 Validator

这个版本至少要支持：

- 输入一个用户目标
- 读取项目文件或外部上下文
- 调用 1 到 2 个工具
- 记录行动日志
- 在一次失败后进行重试

### 第三步：再补底层理论

当你已经做出一个最小版本之后，再回头学习理论会更有效。优先补这些主题：

- prompt orchestration
- tool calling / function calling
- planning
- memory
- eval
- guardrails

### 第四步：最后再研究高级模式

等单 Agent 闭环稳定之后，再研究：

- ReAct
- planner-executor
- reflection / self-critique
- multi-agent coordination
- long-running agent
- checkpoint resume

## 5. 一周可执行学习计划

如果你已经会写代码，但还没有系统做过 Agent，可以按下面的节奏推进。

### Day 1：只看，不做

目标：建立对 Agent 运行轨迹的直觉。

行动：

- 观察 `Codex` 或 `Claude Code` 连续完成 3 到 5 个任务
- 记录每个任务的执行轨迹
- 标注哪些步骤是“理解问题”、哪些是“调用工具”、哪些是“生成结果”

产出：

- 一份你自己总结的执行轨迹模板

### Day 2：拆结构，不追效果

目标：把表象行为拆成模块。

行动：

- 把一个成熟 Agent 的行为映射到模块
- 尝试回答下面这些问题：
  - 输入从哪里进入
  - 计划在哪里生成
  - 工具在哪里注册
  - 状态在哪里保存
  - 输出在哪里校验

产出：

- 一张最小 Agent 结构图

### Day 3：做最小调用闭环

目标：先跑通一个最小 Agent，不追复杂能力。

行动：

- 接入一个模型
- 实现一次标准对话调用
- 让模型稳定输出结构化结果
- 增加一个最简单的工具调用

产出：

- 一个能接收目标并返回结构化结果的最小 CLI 或脚本

### Day 4：加入执行器与日志

目标：从“能问答”升级到“能执行”。

行动：

- 增加 Planner
- 增加 Orchestrator
- 让步骤能顺序执行
- 把中间结果写入日志

产出：

- 一个能按步骤执行任务的最小单 Agent

### Day 5：加入失败恢复

目标：理解 Agent 工程化的起点不是更聪明，而是更稳定。

行动：

- 给工具调用增加超时处理
- 给失败步骤增加重试
- 明确区分可恢复错误和不可恢复错误

产出：

- 一套基础错误处理与重试机制

### Day 6：加入上下文管理

目标：理解 Memory 不是“存更多”，而是“存对的东西”。

行动：

- 区分会话记忆和任务记忆
- 设计简单摘要机制
- 控制上下文长度

产出：

- 一套最小 Memory 裁剪与摘要策略

### Day 7：加入验证与复盘

目标：建立“结果可交付”的判断能力。

行动：

- 定义任务完成标准
- 对关键结果做校验
- 回看一周实现，列出三个最容易失控的点

产出：

- 一份自己的 Agent v1 复盘文档

## 5.7 最值得先学透的 5 个主题

如果你是系统性学习，不要一上来就追多 Agent。优先把这 5 个主题学透：

1. `Planner` 怎么设计
2. `Tool Gateway` 怎么做统一契约
3. `Memory` 怎么做裁剪和摘要
4. `Orchestrator` 怎么做状态机和重试
5. `Validator` 怎么判定“结果可交付”

## 6. 一个推荐的学习路线

### 阶段 1：先做最小可用版本

目标是做出单 Agent 闭环。

建议只保留：

- 一个模型
- 一个 Planner
- 一个 Orchestrator
- 两三个工具
- 一个简单 Memory
- 一个 Validator

你需要能回答：

- 请求是怎么进来的
- 计划是怎么生成的
- 工具是怎么调的
- 状态保存在哪里
- 错了怎么重试

### 阶段 2：补工程能力

开始增加：

- 状态存储
- 日志与 tracing
- 工具注册机制
- Prompt 模板管理
- 成本控制
- 异常处理

这一步的目标是从“能跑”提升到“能稳定跑”。

### 阶段 3：补生产能力

开始增加：

- 权限系统
- 审计日志
- 异步任务
- 队列
- 人工确认
- 自动评估

这一步的目标是从“自己玩”提升到“别人敢用”。

## 7. 判断自己是否还停留在“看懂产品”

如果你已经看了很多演示，但还回答不清下面这些问题，说明你大概率还停留在“看懂产品”，没有真正进入“会做 Agent”的阶段：

- Agent 的状态存在哪里？
- 工具调用失败后怎么恢复？
- 为什么这一步需要计划，而不是直接执行？
- 上下文过长时怎么压缩？
- 如何判断一次执行完成得好不好？

## 8. 当前阶段的明确建议

如果你现在处在“会写代码，但还没系统做过 Agent”的阶段，最优先的事不是继续看更多演示，而是尽快做出一个最小单 Agent。

原因很简单：

- 只观察成熟 Agent，学习收益会很快见顶
- 一旦自己动手实现，很多之前模糊的点会立刻具体化
- 这时再学架构、Memory、Eval、Guardrails，吸收效率会高很多

## 9.2 每一块学到什么程度才算够用

#### LLM Gateway

至少要学会：

- 如何发起一次标准模型调用
- 如何区分 system、user、tool 等消息角色
- 如何控制温度、最大 token、超时
- 如何让模型输出 JSON 或结构化结果

学会的标志：

- 你能稳定拿到结构化输出，而不是每次靠猜格式解析

#### Tool Gateway

至少要学会：

- 如何定义统一工具接口
- 如何做参数校验
- 如何把不同工具结果转成统一返回格式
- 如何处理超时、失败和重试

学会的标志：

- 新加一个工具时，不需要改业务主流程

#### Planner

至少要学会：

- 如何把目标拆成步骤
- 如何给步骤定义输入和输出
- 如何识别哪些步骤需要工具
- 如何在计划中表达依赖关系

学会的标志：

- 你能把一个复杂请求稳定拆成可执行步骤，而不是每次全靠模型即兴发挥

#### Orchestrator

至少要学会：

- 如何顺序执行计划
- 如何保存每步结果
- 如何在失败时重试或中断
- 如何把工具结果继续喂给模型做下一步

学会的标志：

- 任务失败时你能准确知道失败在哪一步，而不是只看到“执行失败”

#### Memory

至少要学会：

- 如何区分会话记忆和任务记忆
- 如何做上下文裁剪
- 如何做阶段性摘要
- 如何避免把无关历史塞进 prompt

学会的标志：

- 对话变长后，Agent 仍然能抓住主线，不明显跑偏

#### Validator

至少要学会：

- 如何定义“完成”的标准
- 如何对工具结果做校验
- 如何判断是否需要二次执行
- 如何把验证失败反馈回执行器

学会的标志：

- 你能解释 Agent 为什么认为结果“可交付”

## 9.3 再补两个关键问题

### 问题 1：规划组件到底选什么，才不是一上来就选复杂？

这确实是一个非常经典的技术选型问题。`ReAct`、`ToT`、`Reflection`、`CoT` 都能参与“规划”，但它们解决的问题并不一样，不能当成同一层面的替代品。

先给结论：

- 如果你是刚开始做单 Agent：优先 `ReAct`
- 如果任务是明显多步、需要探索多条路径：再看 `ToT`
- 如果任务容易首轮做错、需要自检修正：补 `Reflection`
- `CoT` 更像推理展开方式，不是完整工程架构组件

可以把它们理解成四种不同层次的能力：

#### CoT

适合解决的问题：

- 让模型把中间推理展开
- 提高单步复杂推理的稳定性
- 帮助模型先想再答

局限：

- 它本身不管理工具
- 不管理状态
- 不负责外部行动
- 更像“思考方式”，不是“执行框架”

一句话判断：

- 如果你的问题主要是“模型想不明白”，可以先试 `CoT`
- 如果你的问题已经变成“要不要查文件、调工具、继续下一步”，单靠 `CoT` 不够

#### ReAct

适合解决的问题：

- 思考与行动交替进行
- 需要边看结果边决定下一步
- 要接工具、查信息、逐步逼近目标

优点：

- 非常符合 Agent 的最小闭环
- 工程实现成本低
- 很适合作为第一个可运行架构

局限：

- 容易局部最优
- 对超长任务不一定稳定
- 如果缺少状态约束，容易反复试错

一句话判断：

- 只要你的 Agent 要“想一下，然后调一个工具，再继续想”，`ReAct` 往往就是第一选择

#### ToT

适合解决的问题：

- 需要探索多条候选路径
- 中间决策影响很大
- 一条错误路径会导致整体失败

优点：

- 能显式保留多个候选方案
- 比单路径推理更适合复杂规划
- 对搜索类问题更有价值

局限：

- 成本高
- latency 高
- 工程上更复杂，需要定义分支、评分、剪枝

一句话判断：

- 如果任务像“解题搜索”或“方案搜索”，`ToT` 有价值
- 如果你只是做一个能稳定调用工具的 Agent，先别急着上 `ToT`

#### Reflection

适合解决的问题：

- 首轮结果经常不可靠
- 需要执行后复盘
- 需要让模型发现自己的明显错误并修正

优点：

- 对结果质量提升很直接
- 对代码生成、计划修正、回答校验都很实用
- 可以作为执行后补强层，而不是替代主流程

局限：

- 会增加一次甚至多次模型调用
- 如果没有明确校验标准，反思可能变成空转

一句话判断：

- `Reflection` 更适合放在执行后或验证后，而不是拿它当唯一规划器

#### 一个实用选型顺序

对于大多数学习型项目，建议按这个顺序上能力：

1. 先做 `ReAct` 单路径闭环
2. 再加 `Reflection` 做结果复查
3. 确实遇到复杂搜索问题时，再引入 `ToT`
4. `CoT` 作为 prompt 级增强手段使用，不单独当架构答案

如果压缩成一句工程建议：

- `ReAct` 负责“先跑起来”
- `Reflection` 负责“把质量补上去”
- `ToT` 负责“处理高分支搜索问题”
- `CoT` 负责“增强单次推理表达”

### 问题 2：记忆组件、工具组件、行动组件，各自有哪些典型实现 case？

这个问题本质上是在问：Agent 的三个核心中间层，到底应该怎么落地。最有效的学习方式不是背概念，而是看“最小可实现 case”。

#### 记忆组件 case

##### Case 1：会话摘要记忆

目标：

- 对话轮数变多后，保留主线信息

最小实现：

- 保留最近 `N` 轮原始消息
- 更早历史压缩成一段 summary
- 每轮结束后判断是否需要重新摘要

适合场景：

- 通用聊天 Agent
- 助手型 Agent
- 长对话但任务不强依赖精确原文的场景

你能学到什么：

- 上下文裁剪
- 摘要触发时机
- token 预算控制

##### Case 2：任务状态记忆

目标：

- 让 Agent 知道当前任务做到哪一步

最小实现：

- 保存当前目标
- 保存 plan
- 保存每一步执行结果
- 保存失败原因和重试次数

适合场景：

- Coding Agent
- 工作流 Agent
- 多步骤任务执行器

你能学到什么：

- Memory 不只是“聊天记录”
- 状态对象和会话文本要分开
- checkpoint 恢复要依赖结构化状态

##### Case 3：长期知识记忆

目标：

- 让 Agent 记住用户偏好、领域知识、历史项目经验

最小实现：

- 把高价值片段抽取成结构化条目
- 存到向量库或键值存储
- 新任务开始时按语义检索再注入上下文

适合场景：

- 个性化助手
- 项目知识库 Agent
- 跨会话持续工作的系统

你能学到什么：

- 什么信息值得长期保存
- 检索召回和 prompt 注入的边界
- 长期记忆污染怎么避免

#### 工具组件 case

##### Case 1：文件读取工具

目标：

- 让 Agent 读取本地代码或文档

最小实现：

- 输入：文件路径
- 输出：文件内容、是否成功、错误信息
- 增加路径白名单和长度限制

适合场景：

- Coding Agent
- 文档分析 Agent
- 项目问答 Agent

你能学到什么：

- 工具 schema 怎么定义
- 权限边界怎么加
- 返回结果如何统一格式

##### Case 2：搜索工具

目标：

- 让 Agent 获取当前上下文之外的信息

最小实现：

- 输入：关键词
- 输出：标题、摘要、来源链接
- 增加结果数量限制和来源过滤

适合场景：

- Research Agent
- 联网问答
- 文档检索增强

你能学到什么：

- 工具返回不应该直接塞全量原文
- 结果要先压缩再给模型
- “检索”与“回答”最好分层

##### Case 3：执行命令工具

目标：

- 让 Agent 真的做事，而不是只生成建议

最小实现：

- 输入：命令字符串或结构化命令
- 输出：stdout、stderr、exit code
- 增加超时、目录限制、命令白名单

适合场景：

- Coding Agent
- 运维自动化 Agent
- 数据处理流水线

你能学到什么：

- 工具风险远高于普通文本生成
- 执行权限、超时、审计日志必须先设计
- 工具调用结果需要可验证，而不是只看模型解释

#### 行动组件 case

这里的“行动组件”，本质上就是 `Orchestrator` 或 `Action Runner`，负责把计划真正推进下去。

##### Case 1：顺序执行器

目标：

- 按 plan 一步一步执行

最小实现：

- 读取当前步骤
- 判断是直接回答还是调用工具
- 保存结果
- 推进到下一步

适合场景：

- 单 Agent MVP
- 线性任务流

你能学到什么：

- Agent 闭环的最小骨架是什么
- 行动状态如何流转
- 失败点如何定位

##### Case 2：带重试的执行器

目标：

- 工具失败时不要立即整体终止

最小实现：

- 区分可重试错误和不可重试错误
- 对可重试错误设置次数上限
- 每次重试前记录原因

适合场景：

- 网络调用
- 不稳定工具
- 偶发失败的工程系统

你能学到什么：

- “稳定”比“聪明”更先决定可用性
- Agent 工程化从错误恢复开始

##### Case 3：带验证回路的执行器

目标：

- 执行完不是立刻结束，而是先验收

最小实现：

- 先执行任务
- 再调用 validator
- 验证失败时回流给 planner 或 reflection 模块

适合场景：

- 代码生成
- 报告生成
- 高质量交付要求场景

你能学到什么：

- Action 不只是“执行”
- 完整闭环应该包含“执行 -> 校验 -> 再执行”

### 这三个组件，建议按什么顺序学

建议顺序：

1. 先学行动组件，因为它决定闭环能不能跑起来
2. 再学工具组件，因为它决定 Agent 能不能接触外部世界
3. 最后系统学记忆组件，因为它决定系统能不能在复杂任务里持续稳定

原因不是 Memory 不重要，而是：

- 没有行动组件，你只有一个会说话的模型
- 没有工具组件，你只有一个会推理但不能做事的模型
- 没有记忆组件，你有闭环，但闭环会在长任务里逐渐失控
