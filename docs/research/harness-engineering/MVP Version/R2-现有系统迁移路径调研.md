# R2：现有系统迁移路径调研

> 对应 MVP执行总计划.md → R2。目标：冻结迁移策略（渐进改造 vs 并行新建 vs 替换），确定技术选型。

---

## 高密度概括

现有系统是 **LlamaIndex + ReActAgent** 驱动的 Agentic RAG，核心能力是"单次查询内的检索策略自主选择"。研究型 Agent 需要的是"跨轮研究任务自主推进"——两者在状态管理、决策循环和输出形态上都有本质差异。

关键判断：
1. 现有 `AgenticQueryEngine` **不能直接扩展为研究内核**。它是 ReActAgent 的薄包装，没有研究状态、判断形成和收束决策能力。
2. 现有 `RAGService` 的检索能力（vector/hybrid/multi + rerank）**可以作为研究内核的取证工具复用**。
3. 迁移策略推荐 **并行新建 + 接口复用**：研究内核作为新模块（`research_kernel/`）并行于现有 agentic 路径，复用底层检索能力，不改动现有代码。

## 1. 现有系统架构摸底

### 1.1 查询链路

```
用户输入 → RAGService.query()
  ├─ use_agentic_rag=True → AgenticQueryEngine
  │   └─ ReActAgent（LlamaIndex）
  │       ├─ query_processing_tools: analyze_intent, rewrite_query, decompose_multi_intent
  │       └─ retrieval_tools: vector_search, hybrid_search, multi_search
  │   → extract_sources / extract_reasoning → format → RAGResponse
  │
  └─ use_agentic_rag=False → ModularQueryEngine
      └─ QueryProcessor → Retriever → Reranker → LLM → ResponseFormatter → RAGResponse
```

### 1.2 AgenticQueryEngine 能力边界

| 维度 | 现有能力 | 研究内核需要 | Gap |
|------|---------|------------|-----|
| **决策范围** | 单次查询内选择检索策略 | 跨轮研究任务编排 | ❌ 本质不同 |
| **状态** | 无持久状态，每次查询独立 | 研究问题、证据账本、阶段性判断、预算 | ❌ 完全缺失 |
| **反思** | 无 | 证据充足性、知识缺口、证据冲突 | ❌ 完全缺失 |
| **输出** | 答案 + 来源 + 推理链 | 判断 + 证据 + 置信度 + 张力 + 下一步问题 | ❌ 形态不同 |
| **预算控制** | max_iterations + timeout | budget_turns + 边际价值判断 | ⚠️ 需重建 |
| **工具** | 检索工具（3种）+ 查询处理工具（3种） | 同上 + 可能新增 | ✅ 可复用 |

### 1.3 可复用资产

| 资产 | 位置 | 复用方式 |
|------|------|---------|
| **检索能力** | `rag_engine/retrieval/`, `rag_engine/reranking/` | 研究内核的取证工具调用这些现有检索器 |
| **索引管理** | `infrastructure/indexer/` | 直接复用 IndexManager |
| **LLM 工厂** | `infrastructure/llms/` | 研究内核的 LLM 调用走统一工厂 |
| **配置系统** | `infrastructure/config/` | 新增研究内核相关配置项 |
| **日志 + 可观测** | `infrastructure/logger.py`, `observers/` | 直接复用 |
| **RAGResponse 模型** | `rag_api/models.py` | 可扩展或新建 ResearchResponse |
| **ChatManager 会话** | `chat/manager.py` | ⚠️ 不适合直接承载研究状态，但可参考延迟加载模式 |

## 2. 三种迁移策略评估

### 策略 A：渐进改造（扩展 AgenticQueryEngine）

**做法**：在现有 AgenticQueryEngine 基础上增加状态管理、反思循环和判断形成能力。

| 维度 | 评估 |
|------|------|
| 改动量 | 大——需要重写 engine 核心逻辑，ReActAgent 的 tool-calling 范式与 FSM 不匹配 |
| 风险 | 高——改动影响现有 agentic 查询路径，可能引入回归 |
| 依赖 | 继续绑定 LlamaIndex ReActAgent，但 FSM 模式需要的状态图能力 ReActAgent 不原生支持 |

**结论**：❌ 不推荐。ReActAgent 的 tool-calling-in-a-loop 范式与 FSM 的显式状态转换范式本质不同。

### 策略 B：并行新建 + 接口复用

**做法**：新建 `backend/business/research_kernel/` 模块，独立于现有 agentic 路径。研究内核调用现有检索能力作为取证工具，通过 RAGService 新增路由接入。

| 维度 | 评估 |
|------|------|
| 改动量 | 中——新模块独立开发，现有代码不动 |
| 风险 | 低——不影响现有查询路径 |
| 依赖 | 研究内核可选用 LangGraph 或手写 FSM，检索层继续用 LlamaIndex |

**结论**：✅ 推荐。最符合现有架构的分层原则（研究内核是业务层编排模块），且 orient-report.md 的 Scope 决定 2 已选定此方案。

### 策略 C：完全替换

**做法**：用 LangGraph 重写整个查询链路，替换 LlamaIndex 的 ReActAgent。

| 维度 | 评估 |
|------|------|
| 改动量 | 极大——需要替换 AgenticQueryEngine、重写工具接口 |
| 风险 | 高——影响全局，MVP 阶段不值得 |
| 依赖 | 全面转向 LangChain 生态 |

**结论**：❌ 不推荐。违反 MVP 最小改动原则。

## 3. 推荐方案：并行新建 + 接口复用

### 3.1 模块位置

```
backend/business/
  ├── rag_engine/          # 现有，不动
  │   ├── agentic/         # 现有 AgenticQueryEngine，保持不变
  │   └── core/            # 现有 ModularQueryEngine，保持不变
  ├── rag_api/             # 现有 RAGService，新增研究路由
  ├── chat/                # 现有 ChatManager，保持不变
  └── research_kernel/     # ← 新建：研究内核
      ├── state.py         # 研究状态定义（ResearchState）
      ├── engine.py        # 研究引擎（FSM 主循环）
      ├── nodes/           # 各状态节点实现
      │   ├── focus.py     # S1: 问题定焦
      │   ├── plan.py      # S2: 证据计划
      │   ├── gather.py    # S3: 取证（调用现有检索能力）
      │   ├── synthesize.py # S4: 综合判断
      │   ├── converge.py  # S5: 收束决策
      │   └── deliver.py   # S6: 启发式交付
      └── prompts/         # 研究内核专用 prompt
```

### 3.2 接入方式

```python
# RAGService 新增路由逻辑（概念示意）
def query(self, question, mode="standard", ...):
    if mode == "research":
        return self._research_query(question)
    elif self.use_agentic_rag:
        return self._agentic_query(question)
    else:
        return self._standard_query(question)
```

这对应冻结集 #7（快速路径 + 深度研究分级处理）的工程落地。

### 3.3 检索复用方式

研究内核的 `gather.py`（S3 取证节点）通过调用现有检索器完成取证：

```python
# 概念示意
from backend.business.rag_engine.retrieval.factory import create_retriever
from backend.business.rag_engine.reranking.factory import create_reranker

def gather_evidence(state: ResearchState) -> dict:
    """S3: 取证节点 — 调用现有检索能力"""
    retriever = create_retriever(index=..., retrieval_strategy="hybrid", ...)
    nodes = retriever.retrieve(state["evidence_plan"][0])
    # ... rerank, filter, format as evidence
    return {"evidence_ledger": [formatted_evidence]}
```

## 4. 技术选型与 LlamaIndex 共存

### 4.1 如果选 LangGraph

- 研究内核用 LangGraph `StateGraph` 实现 FSM
- 检索层继续用 LlamaIndex（VectorStoreIndex, Retriever, Reranker）
- 两者在不同层共存：LangGraph 在业务编排层，LlamaIndex 在基础设施层
- **不冲突**：LangGraph 管研究流程编排，LlamaIndex 管检索执行

### 4.2 如果选手写 FSM

- 研究内核纯 Python 实现状态机
- 需自行实现：状态存储、条件路由、预算控制、日志追踪
- 优势：零新依赖；劣势：需重新造轮子

### 4.3 依赖影响评估

| 新增依赖 | 影响 |
|---------|------|
| `langgraph` | ~5 MB，依赖 langchain-core |
| `langchain-core` | 可能与现有 LlamaIndex 存在部分 API 重叠，但不冲突 |
| 现有 `llama-index` | 保留不动，继续服务检索层 |
