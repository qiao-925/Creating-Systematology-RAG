# 研究型 Agent MVP 定义与执行计划

## 先说结论

我们现在要做的，不是“更强一点的 RAG”，也不是“更完整一点的报告生成器”。我们要做的是一个研究型 Agent：它能在证据约束下推进问题、形成判断，并把判断转成对用户有启发的结果。

这份计划的目的也很明确：先把第一版产品定义、最小骨架、范围边界和推进顺序说清楚，再进入实现。现在如果急着写代码，后面大概率会把系统做成工具堆砌。

## 为什么这一步必须先做

当前真正的问题，不是“RAG 怎么搭”或者“工具怎么接”。真正的问题是：怎样做出一个能围绕研究问题持续推进，而不是只会给出一轮答案的系统。

如果这件事不先定义清楚，后面很容易反复掉进两条老路：

- 能找资料，但不能形成判断。
- 能生成成稿，但不能给出启发。

所以，这份文档只解决五个问题：

1. 我们到底在做什么产品。
2. 第一版到底要证明什么。
3. 第一版明确做什么，不做什么。
4. 这个系统的最小骨架是什么。
5. 应该按什么顺序推进，才不会过早陷入实现细节。

## 我们到底在做什么

**一句话定义：这是一个以受控自主判断为核心、以高密度知识场为证据支撑、以启发用户为目标的研究型 Agent。**

这里有五个判断先冻结下来：

- 目标是启发，不是总结。
- 核心是判断形成，不是材料拼接。
- 第一版从单 Agent 起步，不预支多 Agent 复杂度。
- 知识库和工具是支撑，不是系统本体。
- 治理和边界不是附属件，而是系统成立的前提。

换句话说，我们不是在造一个“更会搜”的系统，而是在造一个“更会判断”的系统。

## 这个产品为什么值得做

在高密度知识场里，用户真正缺的通常不是更多材料，而是三件事：

- 看清关键张力。
- 在信息不完整时形成更强判断。
- 被带到更值得继续追问的问题上。

这也是为什么“知识库”只是手段，不是目标。用户真正买单的，不是信息量，而是解释力和启发性。

## 第一版到底要证明什么

第一版不用证明系统很全，只要证明三件事成立：

1. 它能围绕研究问题形成过程，而不是只做一次性回答。
2. 它能基于证据形成阶段性判断，而不是只拼接材料。
3. 它的输出能带来启发，而不是只显得完整。

如果要用更直白的话说，MVP 要回答的只有一个问题：这个系统到底能不能把 `问题 -> 取证 -> 综合 -> 判断 -> 启发` 这条链路跑通。

### 成功信号

第一版至少要出现以下信号：

- 面对研究型问题时，输出里有明确判断，而不只是摘要和资料罗列。
- 判断可以回溯到证据，不是脱离材料的空转。
- 输出能揭示关键张力、隐藏关系，或者新的追问方向。
- 系统能在预算内决定何时补证据、何时收束、何时停止。
- 在高风险边界上，系统是受控的，不会无止境自主循环。

### 失败信号

下面这些情况，说明第一版其实没有成立：

- 输出主体仍然是“总结材料”。
- 结论看起来像判断，但说不清依据来自哪里。
- 回答很完整，却没有新视角、新问题或新解释力。
- 系统不断追加查询和工具调用，但判断质量没有明显提升。
- 一旦证据不足，系统就用更长的话掩盖不确定性。

## 第一版做到哪里为止

第一版只做最小必要闭环。只要不直接服务于“证据约束下的判断形成”，就不应该抢先进入。

### 第一版必须做的

- 一个面向研究型问题的单 Agent 主链路。
- 一个能支撑判断形成的高密度知识场证据底座。
- 一个最小可运行的研究内核。
- 一套有限自主边界。
- 一种最小可交付的启发式输出形式。

### 第一版明确不做的

- 多 Agent 协作架构。
- 重型长期记忆系统。
- 大而全的跨领域扩展。
- 把输出模板设计成系统中心。
- 旧系统承接细节。
- 技术栈、目录结构和工程实现细节。

范围控制的原则很简单：MVP 不是为了证明“以后什么都能做”，而是为了证明“最关键的产品命题已经成立”。

## 最小系统骨架

先把最重要的判断说在前面：**研究内核才是系统本体。**

模型、检索、浏览器、数据库、状态存储都很重要，但它们都只是支撑条件。真正决定这是不是研究型 Agent 的，是系统能不能自己完成问题定焦、证据推进、判断形成和收束决策。

最小骨架如下：

```text
┌──────────────────────────────────────────────┐
│ 1. 接入层                                    │
│ UI / Chat / API                              │
└──────────────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────┐
│ 2. 应用层                                    │
│ Agent Service / Session Manager / Router     │
│ 请求接收 / 上下文组装 / 启动研究任务 / 结果交付│
└──────────────────────────────────────────────┘
                      │
        ┌─────────────┼─────────────┐
        ▼             ▼             ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ 3. 研究内核层 │ │ 4. 证据执行层 │ │ 5. 状态层     │
│ 问题定焦      │ │ 取证路由      │ │ 任务状态      │
│ 下一步决策    │ │ 工具调用      │ │ 研究草稿      │
│ 判断形成      │ │ 结果回收      │ │ 证据账本      │
└──────────────┘ └──────────────┘ └──────────────┘
        │             │             │
        └──────┬──────┴──────┬──────┘
               ▼             ▼
        ┌──────────────┐ ┌──────────────┐
        │ 6. 模型层     │ │ 7. 能力层     │
        │ LLM Gateway   │ │ RAG / Search  │
        │ Prompt Policy │ │ Browser / API │
        │ Model Routing │ │ Corpus / DB   │
        └──────────────┘ └──────────────┘
               │             │
               └──────┬──────┘
                      ▼
┌──────────────────────────────────────────────┐
│ 8. 基础设施层                                │
│ State Store / Vector DB / Cache / Queue     │
└──────────────────────────────────────────────┘

外圈持续包裹：
- Guardrails / Budget / HITL / Tracing / Eval
```

这张图真正表达的是四个意思：

- 研究内核决定系统是不是研究型 Agent。
- 证据执行层和状态层决定研究过程能不能持续、可回溯、可收束。
- 模型层和能力层提供智能与取证能力，但不替代研究内核做最终判断。
- 治理外圈必须从第一版就存在，否则有限自主没有可信边界。

## 研究内核到底是什么

如果只用一句话来定义研究内核，那就是：**它负责把证据推进成判断。**

这件事至少要经历六个动作，而且顺序不能乱。

### 1. 问题定焦

先把用户问题收敛成一个可研究的问题。这里要定清楚四件事：目标是什么、范围在哪里、当前约束是什么、什么情况下应该停。

### 2. 证据计划

不是一上来就检索，而是先判断现在缺什么证据，应该优先补哪类证据，哪些证据只是“看上去有用”。

### 3. 取证与校验

系统发起检索、调用工具、回收结果，然后判断这些证据强不强、相关不相关、彼此有没有冲突。

### 4. 综合与判断

这一步的任务不是“把材料整理漂亮”，而是基于已有证据形成阶段性判断，同时保留不确定性、反向证据和悬而未决的张力。

### 5. 下一步决策

系统要决定现在该继续补证据，还是已经可以收束判断，或者应该停止并把控制权交还给人。

### 6. 启发式交付

最终交付的不只是结论，还应该包括三类内容：当前判断成立的关键依据、问题里的关键张力、下一步值得继续追问的方向。

研究型 Agent 的关键不在于“每轮都回答”，而在于“每轮都推进”。真正的推进至少包括三种：

- 把模糊问题推进成可研究问题。
- 把分散材料推进成阶段性判断。
- 把已有判断推进成更有价值的下一步问题。

### 研究内核最少要记住什么

为了让这个闭环能持续，系统至少要持续承载这些状态：

- 当前研究问题与边界。
- 已有证据及其来源。
- 当前阶段性判断。
- 尚未解决的关键张力。
- 候选下一步动作。
- 当前预算与停止原因。

这里的重点不是“记得越多越好”，而是“让研究过程可延续、可审计、可收束”。

### 判断形成最少要满足什么

第一版的判断形成至少要满足四个要求：

- 能区分事实、推断和未知。
- 能在证据不完整时给出阶段性判断，而不是假装已经得出终局结论。
- 能显式处理冲突证据，而不是只保留支持自身结论的材料。
- 能说明为什么现在该继续查、该收束，或者该停止。

## 自主边界怎么控制

这里最容易失控，所以先把边界讲死。

### 系统可以自主做什么

第一版允许系统在边界内自主完成这些动作：

- 把用户问题收敛成可研究问题。
- 规划取证顺序和优先级。
- 调用既定工具补充证据。
- 比较证据、识别冲突、形成阶段性判断。
- 在预算约束下决定继续、收束或停止。

### 系统不应默认做什么

第一版不应默认让系统做这些事：

- 无限制扩展研究范围。
- 为了显得完整而追加低价值取证。
- 在关键证据不足时输出强结论。
- 用更复杂的报告结构掩盖判断薄弱。
- 未经授权就切换任务目标或重写用户问题。

### 哪些场景必须交还给人

以下场景必须触发 HITL，或者明确把决定权交还给用户：

- 用户目标本身有歧义，而且不同理解会明显改变研究方向。
- 结论涉及高风险、高代价或强价值判断。
- 关键证据持续冲突，系统在当前预算内无法合理收束。
- 是否继续投入更多预算，已经变成价值判断，而不是技术判断。
- 系统准备改变研究范围、判断标准或交付目标。

### 停止条件是什么

第一版必须有清楚的停止条件：

- 当前证据已经足以支撑阶段性判断。
- 新增证据的边际价值已经明显下降。
- 预算已经达到上限。
- 关键证据无法获得，继续自主推进的价值不高。
- 已经进入需要人类确认的高风险边界。

没有停止条件的自主，不是能力，只是失控。

## 应该按什么顺序推进

这部分的原则只有一句话：**先冻结核心，再进入实现。**

### 第一步：先冻结研究内核

这一阶段只做一件事，把最小研究闭环定义清楚。到这一阶段结束时，团队必须能清楚描述：一个研究问题进入系统后，会经过哪些步骤、产生哪些状态、在什么条件下继续或停止。

### 第二步：再冻结支撑边界

把研究内核和证据执行层、状态层、模型层、能力层的边界分清。目标是避免任何一层“顺手替代”研究内核做核心判断。

### 第三步：冻结有限自主与治理

把自主边界、预算、停止条件、Tracing、Eval 和 HITL 放到正确位置。目标不是让系统“尽量多做”，而是让系统“只做应该做的那部分”。

### 第四步：冻结 MVP 交付

这一阶段只回答三个问题：

- 什么样的输出已经足够说明“有启发”。
- 输出里哪些部分是必须项。
- 如何做最小评估和回看。

这里要注意，输出很重要，但输出协议不应该反过来主导系统设计。

### 第五步：再进入实现

只有前面四步冻结完成，才进入旧系统承接、技术选型、工程拆分和代码实现。这不是拖慢进度，而是避免把实现建立在错误的问题定义上。

## 当前阶段什么时候算完成

当前这一轮不以“开始写代码”为完成标志，而以下面五件事为准：

1. 产品定义已经清楚，不会再被误解成 RAG 增强版或报告生成器。
2. 研究内核已经清楚，最小闭环可以被完整描述。
3. MVP 范围和非目标已经清楚，不再预支未来复杂度。
4. 有限自主边界已经清楚，知道系统何时自主、何时停下、何时交还给人。
5. 系统骨架已经清楚，后续实现不会因为职责混杂而塌成一团。

## 当前最大的风险和假设

这部分不展开技术细节，只保留当前最重要的判断。

### 主要风险

- “启发”太抽象，最后又回到空泛表达。
- 研究内核定义不清，系统退化成高配检索器。
- 工具和知识能力扩张过早，掩盖主链路问题。
- 停止条件不清，导致低价值自主循环。

### 主要假设

- 单 Agent 足以承载第一版主链路。
- 高密度知识场足以支撑有价值的阶段性判断。
- “启发”可以先用成功信号验证，而不是先追求精确指标。

### 这轮明确后置的内容

以下内容统一后置，不在这轮展开：

- 多 Agent 设计。
- 重型长期记忆。
- 详细状态机。
- 输出协议字段。
- 旧系统承接细节。
- 具体技术栈。
- 代码目录与模块拆分。

## 收束

第一版最重要的，不是把能力堆齐，也不是把架构铺大。第一版真正要先证明的只有一件事：**这个系统能不能在证据约束下形成判断，并把判断交付为对用户有启发的研究结果。**

这件事如果没有成立，其他工作都只是外围优化。  
这件事如果成立，后面的多 Agent、长期记忆、能力扩展和工程放大，才值得继续做。
