# Core Beliefs

> 来源：[Harness Engineering](https://openai.com/index/harness-engineering/) 精髓提取，映射到本项目。
> 这不是规则（规则会过期），是信念（信念指导规则生成）。

---

## Ⅰ. 设计环境

> 让 agent 不仅能读代码，还能运行、观察、验证。

### 1. 运行时可读性

Agent 必须能直接观察到自己的产出。UI、日志、指标对 agent 可读，才能形成闭环。

- **原文**：让应用按 git worktree 启动，Chrome DevTools 接入 agent 运行时，临时可观测栈随任务创建和销毁。
- **本项目**：CLDFlow agent 提取 CLD 后，必须有办法验证结果——不是靠人看，是靠结构化检查（节点数合理性、边连通性、Schema 合规）。FCM 仿真后必须有办法观察收敛曲线。D2D 分析后必须有办法观察杠杆点排序是否合理。
- **行动**：每层输出必须附带自检函数，agent 可调用以验证自身产出。

### 2. 枯燥技术偏好

"枯燥"技术更易被 agent 建模：可组合、API 稳定、训练集覆盖充分。宁可重实现子集也不绕过不透明上游。

- **原文**：不用 p-limit，自实现 map-with-concurrency：100% 测试覆盖、与 OTelemetry 紧密集成、行为完全符合预期。
- **本项目**：优先选择 NetworkX（图操作标准库）、Pydantic（验证标准库）、Python 标准库。避免引入行为不透明的黑盒依赖。如果第三方库的内部行为无法被 agent 推理，优先自实现子集。
- **行动**：技术选型时，"agent 可推理性"是显式评估维度，与功能、性能并列。

### 3. 隔离执行环境

Agent 在完全隔离的环境中运行，任务完成即销毁。避免状态污染。

- **原文**：每个 worktree 启动独立应用实例+临时可观测栈，任务完成即销毁。
- **本项目**：CLDFlow 的每次研究运行应该是隔离的——不依赖前次运行的残留状态。研究状态在运行内维护，运行结束即归档。Phase 1 不做持久化研究记忆（与 orient-report 决定 4 一致）。
- **行动**：Conductor 编排器必须支持运行隔离，运行间无隐式状态共享。

## Ⅱ. 明确意图

> 只执行真正不可变的不变量，边界内允许自主。当一切都重要，什么都不重要。

### 4. 地图优于说明书

AGENTS.md ≤ 100 行做目录，详细知识下沉到结构化 docs/。渐进式披露：小入口 → 按需深入。

- **原文**：巨大指令文件挤掉任务上下文；过多指导=无指导；单文件立即腐烂；难以机械验证。
- **本项目**：当前 AGENTS.md 已较精简（~40行），但 issue-15 有 30 个决策 + 26 个任务 + 大量子文档，agent 无法在一次运行中全部内化。决策和任务应下沉到 `docs/CLDFlow-invariants.md`（不变量）和 `docs/CLDFlow-defaults.md`（默认值），AGENTS.md 只放地图。
- **行动**：AGENTS.md 重写为纯地图模式，指向 core-beliefs + invariants + defaults + architecture。

### 5. 执行不变量，不微管实现

定义严格边界（层级依赖方向、数据格式契约），但边界内允许自由。中央强制边界，本地允许自主。

- **原文**：每个业务域划分为固定层，依赖方向严格验证，仅允许有限边。约束由自定义 linter 和结构测试机械执行。不规定具体实现方式。
- **本项目**：
  - **不变量**（必须执行）：三层依赖方向（前端→业务→基础设施，禁止反向）；CLD 输出必须符合 JSON Schema；CLD→FCM→D2D 流水线不可拆；全自动协作无人工介入。
  - **自由区**（agent 自主选择）：节点归并阈值、冲突分级阈值、FCM 激活函数选择、权重映射档位——这些是实现细节，不是不变量。
- **行动**：不变量编码为 Pydantic 验证 + 结构测试；实现选择只记录默认值，不强制。

### 6. 仓库即唯一事实源

Agent 运行时无法访问的 = 不存在。Slack 讨论、人脑决策 → 必须沉淀为仓库文档。

- **原文**：那次让团队在架构模式上达成一致的 Slack 讨论？如果 agent 无法发现它，它就像迟了三个月入职的新员工一样对其一无所知。
- **本项目**：issue-15 的 30 个决策目前散落在 GitHub issue、对话记录和子文档中。agent 无法在一次运行中全部发现。必须把关键决策沉淀为仓库文档。
- **行动**：所有决策和设计理由必须在 docs/ 中有对应文档。不在仓库中的决策，对 agent 来说不存在。

### 7. 规则升级路径

约束不是一开始就要编码为 lint。有 doc → lint → code 的逐级升级路径。

- **原文**："当文档不够完善时，我们会将规则转化为代码。" 人类品味捕获一次，持续执行。
- **本项目**：
  - **Level 0 - 信念**：`docs/core-beliefs.md`（本文档）——定义"怎么工作"的元规则。
  - **Level 1 - 不变量文档**：`docs/CLDFlow-invariants.md`——5-7 个不可变约束，用自然语言 + 示例表达。
  - **Level 2 - Pydantic 验证**：不变量编码为 Pydantic model 的 validator，运行时强制执行。
  - **Level 3 - 结构测试**：不变量编码为测试用例，CI 强制执行。
- **行动**：规则按验证强度分级。只有被违反过 ≥1 次的规则才升级到 Level 2+。未违反的规则留在 Level 1 即可。

## Ⅲ. 构建反馈回路

> Agent 不是一次性执行完就结束，是自审→被审→修正→再审的循环。

### 8. Agent 困境 = 信号

当 agent 无法推进时，不"更努力"——问"缺什么能力"，补脚手架。

- **原文**：早期进展慢不是因为 Codex 不具备能力，而是因为环境规范不够明确。人类工程师总是追问："究竟还需要什么样的能力，我们又该如何让这个能力对 agent 来说既清晰可读又可强制执行？"
- **本项目**：如果 CLD 提取 agent 反复产出低质量结果，不是换更强模型或加更多 prompt 指令——是问：输入增强够不够？检索到的文档质量如何？Schema 约束是否清晰？缺什么就补什么。
- **行动**：agent 失败时，记录失败模式和缺失能力，作为脚手架改进的输入。

### 9. 自审循环（Ralph Wiggum Loop）

Agent 自审 → 请求 agent 审查 → 响应反馈 → 循环直到满意。agent-to-agent review 是主要审查方式。

- **原文**：指示 Codex 在本地审核其自身更改，请求额外特定 agent 审查，响应反馈，循环直到所有 agent 审核人员满意。
- **本项目**：CLDFlow 的裁判 Agent（D4）只是消解高分歧。但缺少自审循环：提取 Agent 产出 CLD 后，应先自审（结构合理性检查），再送归并器。归并器产出融合 CLD 后，应被评估 Agent 审查。FCM 仿真后，应自检收敛性。
- **行动**：每层 agent 产出后必须经过自审+他审才传递到下游。自审 = 结构化检查函数；他审 = 评估 Agent 或裁判 Agent。

### 10. 品味注入机制

自定义 lint 的错误信息中注入修复指令。人类品味捕获一次，持续执行于每一行代码。

- **原文**：由于这些 lint 是自定义的，我们编写错误信息时会在 agent 上下文中注入修复指令。人类的品味会不断反馈到系统中。审查评论、重构 PR 和面向用户的 Bug 被记录为文档更新或直接编码到工具中。
- **本项目**：Pydantic 验证失败时的错误信息不应只是 "validation error"，应包含修复指引（如 "节点 name 不能为空，请确保每个节点有描述性名称"）。评估 Agent 的评估标准应编码为可执行检查，不只是自然语言描述。
- **行动**：所有验证错误信息必须包含修复指引。所有"品味"判断（如命名规范、文件大小限制）编码为可执行检查。

### 11. 渐进自主

自主性是渐进的：随着测试/验证/审查/反馈/恢复被编码到系统中，agent 才能端到端驱动。

- **原文**：随着越来越多的开发环节被直接编码到系统中——测试、验证、审查、反馈处理和恢复——该仓库最近跨过了一个重要门槛，使 Codex 能够端到端驱动一个新功能。
- **本项目**：不能一开始就期望 CLDFlow 全自动端到端运行。Phase 1 应该是：人给定研究问题 → agent 提取 CLD → 人检查 → agent 继续 FCM → 人检查 → agent 继续 D2D。随着验证和审查机制成熟，逐步减少人工检查点。
- **行动**：自主性分级。Level 0：人驱动每步。Level 1：agent 执行、人审查关键节点。Level 2：agent 端到端、人只审查最终产出。Level 3：agent 端到端、agent 自审。当前目标：Level 1。

### 12. 吞吐量哲学

纠错成本低，等待成本高。短命 PR、flaky test 用后续重跑解决而非阻塞。

- **原文**：在一个 agent 吞吐量远超人类注意力的系统中，纠错成本低，而等待成本高。在低吞吐量环境中这样做是不负责任的。而在这里，这通常是正确的选择。
- **本项目**：当前是低吞吐量环境（3人团队、手动执行），这条暂时不适用。但随着 agent-nightshift 自动化程度提高，应逐步从"阻塞式验证"转向"快速合并+后续修正"。
- **行动**：当前保持阻塞式验证。当 agent 吞吐量显著超过人类审查能力时，重新评估。

## Ⅳ. 跨层元命题

> 纪律在脚手架，不在代码。

### 13. 人类掌舵，智能体执行

人类工作抽象层上移：优先排序、翻译反馈为验收标准、验证结果。不写代码。

- **原文**：人类始终参与其中，但工作的抽象层次与过去不同。我们优先处理工作，将用户反馈转化为验收标准，并对结果进行验证。
- **本项目**：人的职责是——定义研究问题、设定验收标准（什么样的 CLD 算"好"）、验证最终产出。不是写提取 prompt、不是调参数、不是逐行审查代码。
- **行动**：每个任务开始前，人只提供：意图 + 验收标准。agent 负责实现和验证。

### 14. 纪律在脚手架，不在代码

保持代码库一致性的工具、抽象和反馈回路比代码本身更重要。

- **原文**：构建软件仍然需要纪律，但纪律更多地体现在支撑结构上，而不是代码上。工具、抽象和反馈回路变得越发重要。
- **本项目**：当前 5 天（04-10 到 04-15）全在规划，零行核心实现代码。这正是"纪律在脚手架"的反面——我们在用规划代替脚手架。应该开始写代码了，从最小可运行块开始，遇到问题再补能力。
- **行动**：停止规划，开始实现。从 CLD 提取 Agent 的最小可运行版本开始。

### 15. 熵与垃圾回收

Agent 会复制已有模式（包括坏模式）。编码 golden principles 为机械规则，后台持续扫描偏差。技术债 = 高息贷款。

- **原文**：Codex 会复现仓库中已存在的模式——甚至包括不均衡或不够理想的模式。技术债就像一笔高息贷款：不断地以小额方式偿还，总比让债务累积再痛苦地一次解决要好。人类品味一旦被捕捉，就会持续应用于每一行代码。
- **本项目**：当前代码库还小，熵问题不严重。但 CLDFlow 实现后，agent 生成的代码和 prompt 会快速膨胀。需要提前准备：编码 golden principles（如"共享工具优于手写辅助"、"边界验证优于 YOLO 探测"），定期扫描偏差。
- **行动**：实现阶段开始后，建立"doc-gardening"机制——定期检查文档与代码是否一致。
