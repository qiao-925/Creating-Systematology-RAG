# 头脑风暴：项目价值与产品方向

> 2026-04-10 对话记录。状态：**未收敛，阶段性回顾**。

---

## 起点：流程图暴露了问题

画完 Research Agent 的运行流程图后，发现当前设计本质上是：

**搜集材料 → 写总结 → 检查总结和材料像不像 → 不像就再搜**

这是"带质检的检索摘要"，不是"研究"。流程图可以套在任何 RAG 应用上，系统科学在这里只是数据，不是设计。

## 第一轮：把方法论编码进 Agent

**想法**：不只是检索系统科学文献，而是把钱学森的方法论（综合集成研讨厅、从定性到定量综合集成法）应用到 Agent 的设计中。

**关键连接**：钱学森在 1990 年代描述的"综合集成研讨厅"——不同领域专家 + 计算机模型 + 多轮迭代综合——本质上就是人机协作研究的原型。30 年后技术终于能实现这个构想。

**结论**：方向有意思，但还没完全通。

## 第二轮：让系统结构可见

**想法**：产品价值不是"帮你得到答案"，而是"帮你看见问题背后的系统结构"（子系统、反馈回路、杠杆点）。

**质疑**：通用模型（Claude Opus 4.6 thinking）在结构化分析上已经很强。不确定专用工具在这个方向上能否胜出。

## 第三轮：实验场定位

**共识**：项目最初就是奔着学习技术做实验去的。

**垂直 Agent 的优势确认**：
- 知识可信度（RAG 不幻觉）
- 方法论一致性（不需要用户知道怎么 prompt）
- 可评估性（结构化流程可度量）
- 可复现性（同一问题可比较）

**定位**：以系统科学为载体的垂直 Agent 架构实验场。

## 第四轮：知识不过期

**关键洞察**：技术会不断迭代，但知识不会过期。

读创建系统学的过程给了很多启发。项目名"Creating Systematology"——重点是 **Creating**，不是 Searching 或 Using。通过构建 AI 系统来重新创建对系统学的理解。

**编码即学习**：要造 RAG 就得梳理知识结构。要设计工具就得想清楚研究分几步。要写评估指标就得定义什么叫好的输出。代码不容许模糊，逼你把每个"大概懂了"变成精确定义。

## 第五轮：创造空间 vs 消费产品

**闪现**：如果产品不是让用户用 Agent，而是让用户造 Agent——创造过程本身就是价值？

**立刻被质疑**：市面上"创建 Agent"的平台很多（Dify、Coze、GPTs）。这个方向像是红海。

## 第六轮：Dify/Coze 的结构性局限

分析了这类平台的 5 个根本限制：

1. **配置 ≠ 设计**——不能定义新的积木类型（如类型化状态模型）
2. **工作流 ≠ 方法论**——固定流程图无法表达基于状态的动态决策
3. **知识库是死的**——只能被检索，不能影响 Agent 行为
4. **没有评估层**——跑完就跑完，无法自我修正或度量质量
5. **所有 Agent 共享同一个认知结构**——无法定义领域特定的认知流程

**类比**：Dify/Coze 之于 AI Agent = 低代码平台之于传统软件。80% 场景覆盖得快，20% 怎么都做不到。

**发现的空白**：AI Agent 领域缺少成熟的"框架"中间层（类似 Rails 之于 Web 开发）。LlamaIndex/LangChain 提供组件但不提供 opinion。

## 第七轮：MethodologyPack 概念

**想法**：把领域方法论封装成可插拔的标准化组件（MethodologyPack），包含工具集 + 状态模型 + 评估标准 + prompt + 知识结构。通用层处理 Agent Loop/LLM/可观测/护栏，领域层通过 Pack 插入。

**与 Skill 的区别**：Skill 是给 LLM 的建议（文本），MethodologyPack 是给 Agent 的骨架（代码结构 enforce）。

**评估**：概念有吸引力，但太天马行空，技术不确定性大，偏社区/生态方向。

## 第八轮：收敛——需要调研

**共识**：纯脑暴不够，缺事实依据。需要做一轮竞品/赛道调研再决定方向。

**调研三问**：

3. 通用模型的天花板在哪？Claude/ChatGPT Deep Research 做不好什么？

## 第九轮：竞品调研 + 收敛

详见 `docs/competitive-landscape-2026-04-10.md`。

**赛道空白**：
- 全部学术研究工具仅支持英文
- 无人做方法论驱动的研究
- 无人做研究过程评估

**收敛：三支柱定位**

> **领域定制 · 可审计 · 评估反馈**

1. **领域定制**：领域知识库 + 领域方法论编码（工具/状态/流程）+ 领域评估标准。护城河在"领域 × 工程"交叉能力，不在技术栈。
2. **可审计**：输出不是黑盒答案，而是可检查每一步的推理过程。解决的是信任问题。
3. **评估反馈**：内建质量度量 + 运行时回流。Agent 自知输出质量，用户可见评估结果。整个赛道独有。

**项目定位（草案）**：

> 以系统科学为方法论内核的深度研究 Agent，输出可审计、可评估、可复现的结构化研究。

## 第十轮：收敛完成 ✅

### 底层逻辑：概率 vs 确定性

LLM 是概率机器。Skill 在概率空间里提高正确行为的概率。架构在确定性空间里 enforce 关键约束。

领域定制的本质：**在概率的海洋里，插入确定性的岛屿。**

- Skill 层（软约束）：引导 LLM 推理方向 → 概率负责创造力
- 架构层（硬约束）：状态/评估/审计 → 确定性负责可靠性

两层叠加 = 完整系统。差异化在于"我们有两层而别人只有一层"。

### 最终三支柱

> **领域定制 · 可审计 · 评估反馈**

| 支柱 | 确定性保证 | 对应赛道空白 |
|------|-----------|------------|
| 领域定制 | 方法论步骤和状态模型是确定的 | 无人做方法论驱动的研究 |
| 可审计 | 每一步的记录是确定的 | 无法复现研究过程 |
| 评估反馈 | 质量度量是确定的 | 无人做研究过程评估 |

### 确认的项目定位

> 以系统科学为方法论内核的深度研究 Agent，输出可审计、可评估、可复现的结构化研究。
> 核心竞争力：领域×工程交叉能力 + 概率-确定性双层架构。

**状态：方向已定 ✅**
