# 简洁优于复杂的工程决策

**日期**：2025-10-09

## 背景

在 Windows-Make 工具配置任务中，最初想同时支持多种运行方式（Makefile、PowerShell、Bash、Batch），但用户做出了关键决策。

## 核心洞察

**"Makefile 是更广泛使用的，从工程实践的角度来看，是否用 Makefile 是最合理的"**

### 用户的思考层次

1. **生产环境** → Linux 为主
2. **CI/CD** → 使用 make 命令
3. **团队协作** → make 是通用语言
4. **开源标准** → 几乎所有项目都有 Makefile

### 最终决策

> "我觉得暂时先只保留 Makefile 一种吧，不想把这件事情变得很复杂"

**执行动作**：
- ✅ 删除所有脚本文件（run.ps1、run.sh、run.bat）
- ✅ 只保留 Makefile
- ✅ 删除相关文档（WINDOWS_GUIDE.md）

## 核心原则

**简洁优于复杂**

- 不要为了"支持所有场景"而增加复杂度
- 选择最广泛接受的标准
- 保持工具链的统一性
- 特殊场景可以通过文档说明解决

## 实践启示

在技术选型和工具设计时：
1. **先问"哪个是标准？"** - 而非"哪个更酷？"
2. **考虑生产环境** - 开发环境可以妥协，生产环境不能
3. **优先通用性** - 团队协作和开源项目需要统一语言
4. **拥抱约束** - 有时限制选项反而能降低复杂度

**这种工程思维，是从全局视角看问题，而非局部优化。**
